# 摘要

手机智能体的核心竞争力正在从“即时对话能力”转向“长期个性化服务能力”，而长期个性化服务的基础正是可持续、可检索、可治理的记忆系统。OpenClaw 官方文档显示，其记忆并非隐藏在模型内部，而是以 MEMORY.md 和 memory/YYYY-MM-DD.md 等明文工作区文件为核心，通过 memory_search、memory_get 以及内建 SQLite/向量混合检索机制实现持续记忆与召回；同时，OpenClaw 也明确指出工作区只是默认当前工作目录（cwd）而非天然硬隔离，安全模型默认建立在“单一可信操作者边界”之上，而不是对抗式多租户隔离边界。由此可见，若要将此类手机智能体应用于个人、办公与第三方协作场景，必须把记忆隐私保护、访问控制、场景隔离与合规共享作为一体化问题来研究。

从制度与基础设施角度看，国家数据局行动计划已将企业、行业、城市、个人、跨境可信数据空间纳入统一推进框架，并明确提出使用控制、数据沙箱、隐私计算、可信执行环境、合规管理和履约存证等能力要求；IDS-RAM 进一步强调数据主权与 使用策略执行（usage policy enforcement），即数据提供方在共享后仍应对数据使用方式保持可控。对手机智能体而言，这意味着“个人记忆”不能仅被视为提示上下文的一部分，而应进一步被抽象为一种可分类、可授权、可审计、可撤销的数据资产。

基于上述背景，本文围绕四个问题展开研究：第一，如何对个性化记忆进行隐私分级分类；第二，如何实现记忆片段的安全存储与细粒度访问控制；第三，如何在跨设备场景下以差分隐私与安全聚合降低同步风险；第四，如何把 OpenClaw 的工作区机制与可信数据空间的数据沙箱结合起来，形成个人、办公、第三方场景下的隔离边界与合规共享路径。本文最终提出一套“本地优先、策略前置、分层存储、用途约束、审计留痕、跨域受控”的总体技术路线。

**关键词：** 手机智能体；OpenClaw；个性化记忆；可信数据空间；使用控制；数据沙箱；差分隐私；跨设备同步

# 一、绪论

## 1.1 研究背景

智能体（Agent）正在经历从无状态对话到持久化有状态智能的关键转变。个人大语言模型智能体（Personal LLM Agents） 综述把这类系统定义为“与个人数据和个人设备深度整合、用于个人协助的 基于大语言模型的智能体（LLM-based agents）”，并将其能力、效率与安全视为共同决定系统可用性的核心因素；另一篇 智能体记忆 综述指出，支持智能体与环境长期交互的关键组件正是 记忆模块（memory module）。这意味着，在手机智能体场景中，记忆不再只是“聊天记录的延长线”，而是任务连续性、偏好适配、主动建议与长期协作的能力底座。

OpenClaw 的实现进一步把这种趋势工程化：它将长期记忆写入 MEMORY.md，将日常观察与上下文写入 memory/YYYY-MM-DD.md，并在会话压缩前执行自动 记忆刷新（memory flush），以减少压缩带来的上下文丢失；当配置了 向量服务提供方（embedding provider） 时，memory_search 使用向量相似度与关键词匹配的混合检索。换言之，OpenClaw 已经提供了一个相当清晰的“外显持久化记忆”范式，这使得相关隐私与合规问题具有可观测、可建模、可控制的技术对象。

但正因为记忆变成了“可持久化、可索引、可调用”的对象，其风险也随之放大。OpenClaw 文档明确强调，工作区应被视为记忆的一部分并保持私密；它同时说明工作区不是硬沙箱，若未启用沙箱（sandbox），绝对路径仍可能触达宿主机其他位置。安全文档则进一步说明，OpenClaw 默认假设的是“单一可信操作者边界”，不把单个共享网关（gateway） 视为敌对多租户安全边界。对手机智能体而言，这意味着只靠“有记忆能力”远远不够，必须追加严格的隔离、授权、同步与共享治理机制。

## 1.2 研究目的与意义

本课题的核心目标，不是单纯提高记忆检索命中率，而是在“更懂用户”和“更少暴露用户”之间建立一套可验证、可审计、可共享的技术路径。国家数据局行动计划已将“个人可信数据空间”列为积极探索方向，并提出应在尊重个人意愿、保护个人权益的前提下，探索依场景授权许可的个人数据转移流动和开发利用服务；同时，文件还把使用控制、数据沙箱、隐私计算、可信执行环境和大模型融合创新列为重点方向。这为“个性化记忆资产化而又受控流通”的研究提供了明确政策空间。

本课题的现实意义体现在三点。其一，对个人用户而言，研究结果可以提升其对智能体记忆系统的可理解性与可控性，降低“越用越懂我”转化为“越用越暴露我”的风险。其二，对企业场景而言，个人域与工作域记忆的隔离，是推动手机智能体进入办公与协作场景的基础。其三，对可信数据空间场景而言，若记忆片段可以被规范化为带有用途、权限、生命周期和审计信息的数据对象，它们就可能从“只服务单个智能体的私有上下文”升级为“可在受控环境中共享利用的数据资产”。

本课题的定位是在手机智能体场景下，兼顾OpenClaw类持久化记忆系统与可信数据空间隐私计算框架的交叉融合，研究个性化记忆的隐私保护管理与合规共享机制。课题旨在探索如何将个人记忆数据转化为可在可信数据空间中安全流通的数据资产，支持用户自主控制记忆数据的共享范围和使用目的。

## 1.3 研究范围

本文讨论的“记忆”特指外显持久化记忆，即原始记忆片段、摘要记忆、向量表示（embedding）、索引、画像标签及其授权和共享元数据，而不涉及基础模型参数中潜在吸收的隐式知识。本文研究对象也以手机智能体为中心，重点覆盖个人场景、办公场景和第三方应用接入场景，不延伸到大模型底座训练安全、广告推荐、跨境合规实务细则等更广泛议题。这个范围界定与OpenClaw当前文档中“记忆写盘—索引—检索—工作区—沙箱”的工程边界是对齐的。

# 二、研究现状

## 2.1 OpenClaw记忆、工作区与安全模型

OpenClaw 的 Memory Overview 文档说明，其记忆通过在智能体工作区（agent workspace）中写入 Markdown 文件实现。模型“记得”的内容来自磁盘上被保存的记忆，而不是隐藏在模型内部的不可见状态。其内建记忆引擎会把这些记忆文件切块后写入按智能体划分（per-agent）的 SQLite 数据库，并支持 FTS5 全文检索、向量检索与混合检索，对中日韩文本还提供三元切分支持。这样的设计有两个直接含义：一是记忆对象边界清晰，便于开展分类分级和加密治理；二是检索层面对的不只是文本文件，还包括切块、向量表示和倒排索引，因此隐私保护不能只盯住原文。

在工作区与运行边界上，OpenClaw 将工作区（workspace）定义为智能体的 home，建议将其视为记忆的一部分并保持私密；但它同时强调，工作区只是默认当前工作目录，并不是强隔离沙箱（sandbox）。只有启用沙箱，并将工作区访问权限设为 "none" 或 "ro" 时，工具操作才会真正被限制在沙箱工作区或只读挂载空间中。这说明“工作区隔离”与“执行隔离”并不是同一件事：前者更偏向逻辑组织边界，后者才是真正的访问边界。

在安全假设上，OpenClaw 的公开文档表述得很明确：它默认采用个人助理信任模型（personal assistant trust model），即一个网关对应一个可信操作者边界，而不是在单一共享网关上提供敌对多租户隔离。对混合信任或对抗式用户场景，文档建议拆分信任边界，例如使用独立网关、独立凭据，最好再配合独立操作系统用户或独立主机。这一前提对本课题十分关键，因为它意味着办公、个人和第三方三类记忆边界若要真正可靠，不能只依赖应用层路由或智能体标识，还必须结合更强的隔离与策略执行。

## 2.2 可信数据空间、数据主权与使用控制

国家数据局行动计划将可信数据空间定义为基于共识规则、联接多方主体、实现数据资源共享共用的数据流通利用基础设施，并提出到 2028 年建成 100 个以上可信数据空间，分类推进企业、行业、城市、个人、跨境可信数据空间建设和应用。对本课题最直接相关的是两点：一是文件明确提出稳慎探索个人可信数据空间；二是文件要求在个人权益保护和个人意愿尊重的前提下，提供依场景授权许可的个人数据转移流动和开发利用服务。

在能力层面，行动计划要求构建接入认证、空间资源使用合约、履约行为存证、全程溯源等能力，并将使用控制、数据沙箱、隐私计算、可信执行环境、密态计算等列为可信管控技术。文件还给出了“使用控制”“隐私计算”“数据沙箱”的明确定义：使用控制是通过预先设置条件形成控制策略并动态决定数据操作许可或拒绝；隐私计算是在不泄露原始数据前提下实现“可用不可见”；数据沙箱是一个应用层隔离环境，允许在安全受控区域内对数据进行分析处理。这些定义几乎可以直接映射到手机智能体的记忆授权、同步与共享场景。

国际数据空间参考架构（IDS-RAM）和相关知识库从数据空间架构角度补足了“共享后仍可控”的逻辑。IDS-RAM 强调数据主权（data sovereignty），即数据使用控制权应保留在数据收集、存储和提供者手中；使用控制（usage control）则被界定为访问控制（access control）的延伸，关注的是“拿到数据之后还能做什么、不能做什么”，并通过把使用策略附着到交换数据上，在运行时持续限制数据被处理、聚合和转发的方式。这对本文的启发是：手机智能体中的记忆共享不能只做“能否读取”的访问控制，还必须进一步约束其是否可用于画像、是否可转发第三方、是否可长期保存以及是否可跨域再用。

## 2.3 隐私、安全与同步研究

围绕手机智能体记忆管理的研究正在快速展开。个人大语言模型智能体（Personal LLM Agents）综述指出，这类系统与个人数据和个人设备深度融合，因此能力、效率与安全需要被同时考虑；另一篇智能体记忆综述指出，智能体记忆的设计与评测仍高度分散，业界尚缺统一的设计范式和评估标准。这意味着，手机智能体记忆管理目前仍处在“需求明确、方法涌现、规范不足”的阶段，本课题既有现实价值，也有方法论空间。

在风险侧，检索增强生成成员推断（RAG membership inference）研究表明，攻击者可能通过观察系统输出，推断某段文本是否存在于检索数据库中；AgentPoison 则展示了通过毒化长期记忆或检索知识库来攻击大语言模型智能体的可能性，在三类真实智能体上平均攻击成功率较高，而正常任务性能几乎不受影响。因此，对记忆系统而言，“泄露”与“污染”必须并行治理：前者关注记忆是否被看见，后者关注记忆是否被操纵。

综合文献分析，现有研究存在以下空白：（1）在手机智能体场景下，缺乏针对记忆数据全生命周期的隐私保护与合规共享系统性方案；（2）OpenClaw工作区隔离机制与数据沙箱技术的结合在学术文献中尚属空白；（3）记忆数据如何作为“数据资产”在可信数据空间中合规流通，缺乏标准化的技术架构。有研究指出，Agent记忆可以作为代理经济中的经济商品，但前提是买方能够验证其真实性、有付出和兼容的执行上下文——然而目前这些记忆仍保持私有且不可转让。

# 三、问题定义、需求分析与威胁模型

## 3.1 个性化记忆的数据对象

从系统实现角度，手机智能体的个性化记忆至少应拆分为六类对象：原始记忆片段、摘要记忆、向量表示（embedding）、检索索引、画像标签、推断结果。OpenClaw 的记忆系统已经明确包含原始 Markdown 记忆文件、切块（chunking） 后的检索单元以及按智能体划分（per-agent） SQLite 索引；而一旦引入个性化服务，系统通常还会产生偏好标签、重要性评分、生命周期状态等派生元数据。也就是说，“记忆”并不等于“一段文本”，而是一个由内容、表示和控制信息共同构成的复合对象。

从生命周期看，一条记忆会经历采集、抽取、写入、索引、召回、复写、共享、删除等多个阶段。国家数据局文件对使用控制、溯源和全程治理的强调，恰好说明不能只保护存储态数据，而要把产生、存储、计算、应用、销毁全过程纳入治理。本文据此将记忆治理目标定义为“全生命周期、全表示形态、全场景用途”的统一治理。

## 3.2 典型使用场景

本文聚焦三类场景。第一类是个人场景，目标是提升私人助理能力，如偏好记忆、日程习惯、长期任务承接。第二类是办公场景，目标是让智能体持续理解工作上下文、会议事项与协作惯例，但必须压缩个人信息暴露面。第三类是第三方应用接入场景，例如外部 App、插件、企业服务或可信数据空间内其他主体请求使用部分记忆数据。这三类场景的共同点是都需要记忆；不同点则在于共享范围、使用目的、审计强度和隔离要求明显不同。行动计划中对个人、企业、行业、城市等不同可信数据空间类型的分类推进，也支持这种“按场景分层治理”的设计思路。

## 3.3 威胁模型

本文将威胁分为四层。第一层是存储与检索层威胁，包括本地文件泄露、索引泄露、向量表示（embedding） 反演、成员推断、检索枚举。第二层是运行时威胁，包括 提示词注入（prompt injection）、工具链越权、日志泄露、缓存残留和工作区越界访问。第三层是记忆内容威胁，包括外部导入内容污染、知识库投毒、跨场景串扰、错误记忆放大。第四层是共享与同步威胁，包括用途漂移、权限蔓延、授权难撤回、跨设备同步中原始信息泄露。前述 RAG membership inference 与 AgentPoison 研究分别为第一层和第三层威胁提供了直接证据。

## 3.4 合规边界

从合规边界看，行动计划明确要求可信数据空间参与各方遵守《网络安全法》《数据安全法》《个人信息保护法》等法律规定，落实数据分类分级保护、动态感知、风险识别、监测预警、应急处置和治理监管要求。这意味着，对手机智能体记忆系统而言，合规不是一个“上线后再补”的外部流程，而必须体现在分类分级、最小必要、用途限定、授权许可、动态控制、审计留痕和删除撤回等设计中。本文因此把“合规共享”具体化为：用户可知、用户可控、策略可执行、过程可留痕、结果可追责。

# 四、关键技术方案研究

第 3 章已经完成了问题定义、威胁模型与合规边界分析。本章进一步面向手机智能体个性化记忆治理的核心环节，提出从记忆分级分类、分层存储与加密访问控制、跨设备差分隐私同步，到工作区隔离与可信数据空间数据沙箱融合的一体化技术方案。其基本思路是：首先，通过面向向量检索场景的隐私分级分类，明确不同记忆对象的治理属性；其次，通过分层存储、分级加密、细粒度访问控制和使用控制，将分类结果转化为可执行保护策略；再次，通过联邦式更新、差分隐私和安全聚合控制跨设备同步中的隐私损失；最后，通过 OpenClaw 工作区与沙箱机制，以及可信数据空间的数据沙箱和数字合约能力，建立个人域、工作域和第三方域之间的双层边界与受控共享路径。四节共同形成“先分类、再保护、后同步、受控共享”的完整技术闭环。

本章内部逻辑关系如下：第 4.1 节解决“记忆是什么、应如何分级分类”的问题，第 4.2 节解决“分类后的记忆如何存储、加密与访问”的问题，第 4.3 节解决“哪些记忆可以跨设备同步，以及如何控制同步中的隐私损失”的问题，第 4.4 节解决“高敏记忆如何在跨场景、跨主体条件下进入受控执行环境而不自由流动”的问题。四节共同形成“先分类、再保护、后同步、受控共享”的完整技术闭环。

## 4.1 基于向量检索的记忆隐私分级分类

### 4.1.1 理论基础与设计目标

要实现手机智能体记忆数据的可控共享，首要前提是对记忆进行可执行、可解释、可审计的分级分类。对 OpenClaw 一类具备持久化记忆能力的系统而言，记忆已经不再只是普通对话文本，而是由原始片段、摘要片段、向量表示（embedding）、检索索引、调用上下文与授权元数据共同构成的复合对象。因此，隐私分级分类不能只回答“哪些内容敏感”，还必须进一步回答“哪些内容可以长期保留、哪些内容可以被跨会话召回、哪些内容可以进入向量索引、哪些内容可以跨设备同步、哪些内容可以在可信数据空间中受控共享”。

OpenClaw 官方文档表明，其长期记忆主要以 MEMORY.md 中的持久事实、偏好和决策形式存在，日常上下文和观察写入 memory/YYYY-MM-DD.md，并通过 memory_search、自动记忆刷新（记忆刷新（memory flush））以及实验性的 Dreaming/Promote 机制在短期上下文与长期记忆之间形成迁移和提升链路。由于其记忆召回已经采用混合检索机制，分类不能仅面向原文内容，还必须面向检索表示和召回过程。也就是说，一条记忆即使没有直接暴露在原文层面，只要已经进入向量索引并可被相似查询召回，就已经具有新的隐私风险与治理需求。

从治理视角看，成熟的可信 Agent 记忆系统必须显式区分不同层级的记忆对象，因为不同层级在持久性、敏感性、可扩散性和治理要求上差异显著。用户特定偏好、私人历史、长期决策模式通常属于高敏对象；当前会话上下文虽持久性较低，但仍可能包含短时敏感信息；系统配置与公共知识则通常具有较低保护要求。OWASP 对 LLM/GenAI 风险的分类也表明，上下文注入、过度共享、敏感信息披露与不安全插件设计会使记忆内容从一个会话或场景扩散到另一个会话或场景。因此，记忆分类的目标不是“静态打标签”，而是为后续的存储、授权、同步、共享和审计提供统一的策略入口。

基于此，本文将本节的设计目标定义为三点：第一，形成一套可落地的记忆隐私分级分类框架；第二，使分类结果能够直接作用于向量索引、存储加密、访问控制、跨设备同步和可信共享策略；第三，使该框架兼顾合规解释责任与工程可实施性。

### 4.1.2 OpenClaw 记忆形态到治理层级的映射

严格来说，OpenClaw 官方直接定义的是长期记忆文件与 daily notes 等持久化记忆载体，而不是一套正式命名的“三层记忆架构”。因此，本文将“短期日志—近端会话—长期知识”理解为一种面向治理的映射视图，用于描述不同记忆对象在保护要求上的差异。

第一层是短期活动记忆层。该层主要对应当前 session 中尚未固化的交互上下文、工具调用结果和临时推理内容，持久性最低、时效性最强。它默认不属于长期持久化对象，但由于其直接参与当前生成过程，仍可能在会话中暴露敏感信息，因此需要避免临时上下文误进入后续推理链路或被不当地刷写进长期记忆。

第二层是近端会话记忆层。该层主要对应近期 daily notes、压缩摘要和近期可自动加载的上下文片段。其持久性高于纯会话态，且已经具有跨轮、跨天召回能力，因此治理要求高于短期活动记忆。由于这部分内容常包含未完全沉淀的观察、任务过程和局部判断，既要防止其被长期错误固化，也要防止其因检索便利而被跨场景误用。

第三层是长期知识记忆层。该层主要对应 MEMORY.md 中的 durable facts, preferences, and decisions，即相对稳定的用户事实、偏好和决策模式。此类内容一旦进入长期记忆，将在未来多轮、多任务和多场景中持续影响智能体行为，因此其敏感度通常最高，也是个性化与隐私保护矛盾最集中的部分。本文认为，这一层应成为后续强保护、细粒度授权和受控共享的重点对象。

这一治理映射的意义在于，它把“是否长期保留”“是否可跨会话召回”“是否已进入向量索引”“是否可跨设备同步”这些问题统一起来，使后续分级分类不再只依赖文本本身，而能结合记忆在系统中的实际位置与传播路径进行判断。

### 4.1.3 五维分类模型

在分类维度设计上，本文建议采用五维分类模型，即从内容敏感度、来源可信度、使用目的、场景域和生命周期五个方面共同刻画记忆对象。与只基于 PII 或关键词的单轴分类不同，这种五维模型更符合手机智能体记忆的实际治理需求。

第一维是内容敏感度。该维度用于区分普通偏好、重要事务、敏感身份信息、高敏感财务/健康/工作机密等不同类型的内容风险。它回答的是“这条记忆本身有多敏感”。

第二维是来源可信度。该维度用于区分用户显式输入、系统总结推断、外部导入内容、第三方服务回填结果等不同来源。来源不同，错误风险、污染风险和授权基础也不同。用户明确表达的偏好，与模型根据若干行为自动推断出的偏好，在治理上不应等价对待。

第三维是使用目的。该维度用于区分个性化服务、任务连续性、跨设备同步、外部共享等用途。相同内容在不同使用目的下的治理要求可能完全不同。例如，一条偏好记忆在本地个性化推荐中风险较低，但若用于对外共享或第三方建模，其风险级别应显著上调。

第四维是场景域。该维度用于区分个人域、工作域、第三方域、可信空间共享域等运行环境。它回答的是“这条记忆允许在什么边界内被使用”。该维度尤其适用于解决个人场景与办公场景记忆串扰的问题。

第五维是生命周期。该维度描述短期、中期、长期和待删除状态。生命周期越长，记忆被反复召回、传播和误用的概率越高；进入待删除或撤销共享状态的对象，则应被优先限制召回与同步。

这五个维度共同把“记忆分级分类”从单纯的内容判断，提升为面向系统治理的综合判断。其中，内容敏感度决定“记忆危险不危险”，来源可信度决定“记忆靠不靠谱”，使用目的决定“为什么要用”，场景域决定“在哪里能用”，生命周期决定“还能用多久”。五维模型的价值在于，它为后续的策略映射提供了结构化输入，使分类结果可以直接衔接存储、检索和共享控制，而不是停留在文本标签层面。

### 4.1.4 敏感度量化与联合判定方法

在实现层面，本文建议采用“规则引擎 + 语义模型 + 元数据治理”的联合判定方法。其核心逻辑是：规则引擎负责识别显式敏感项和强约束情形，语义模型负责识别隐含语义风险和场景语义，元数据治理负责把判定结果转化为后续可执行策略字段。

首先，在数据摄入阶段执行敏感实体检测。该阶段可通过规则匹配与 NER/Transformer 类模型结合，对姓名、手机号、邮箱、身份证号、地址、账号、设备标识符等显式敏感实体进行识别与标注。OWASP 对敏感信息披露的风险说明，以及 RAG 隐私防护相关研究，都支持在 ingestion 早期尽量完成敏感信息识别、最小化和必要的 redaction，以降低后续进入索引和生成链路的暴露面。

其次，执行语义敏感度分析。仅依赖显式 PII 识别是不够的，因为很多高风险记忆并不直接包含强标识符，而是通过“行为模式 + 场景描述 + 关系线索”的组合暴露用户身份与特征。例如，稳定的通勤地点、固定的交易习惯、特定工作项目或家庭成员关系，即使不出现身份证号，也仍然可能构成高风险记忆。因此，语义模型应重点识别隐含偏好、稳定行为模式、健康与财务语义、职位与项目语义，以及多线索组合后的可重识别性。

再次，必须把检索暴露度纳入量化框架。由于 OpenClaw 已使用 混合检索（hybrid search），记忆进入索引后，其风险不再等同于普通本地文本。近期关于 RAG membership inference 的研究表明，攻击者可能通过观察生成结果或构造提示，推断某条样本是否存在于外部检索数据库中；这说明仅仅“把数据变成向量”并不会天然消除隐私风险。

基于此，本文为每条记忆计算一个综合敏感度得分S：

$$S=w_{pii}S_{pii}+w_{sem}S_{sem}+w_{ret}S_{ret}+w_{scope}S_{scope}+w_{persist}S_{persist}$$

其中，$S_{pii}$表示显式敏感实体得分，$S_{sem}$表示语义敏感度，$S_{ret}$表示检索暴露度，$S_{scope}$表示场景与传播范围风险，$S_{persist}$表示持久化和生命周期风险；w为根据场景设定的权重。与传统只看关键词或只看实体数量的方法相比，这一评分框架把内容风险、检索风险和传播风险统一起来，更符合向量检索型记忆系统的实际需求。

### 4.1.5 四层隐私分级体系

在五维分类模型和综合评分的基础上，本文提出四层隐私分级体系，用于统一后续的存储、访问、同步和共享策略。

- **L0（公开/低敏）**：不包含可识别个人信息，不涉及稳定偏好、关键行为模式、工作机密或高风险场景，主要用于一般性检索增强和低风险提示补充。
- **L1（一般敏感）**：包含轻度个性化线索，如普通偏好、常规任务习惯、一般上下文信息，可进入向量索引，但需绑定基础访问控制和最小化共享策略。
- **L2（高敏）**：包含显式 PII、较强身份线索、位置/关系/财务/工作上下文等内容，允许受限索引和受限召回，并默认启用更强的加密、审计和用途约束。
- **L3（核心敏感）**：包含高敏个人信息、关键决策模式、跨场景身份映射、工作机密，或多重线索组合后具有高度可重识别性的内容，原则上不进入普通向量召回通道，仅允许在更高强度隔离或受控沙箱环境中按策略使用。

### 4.1.6 分级分类实现流程

本文建议的实现流程包括六个步骤。

1. **记忆片段提取**。从 MEMORY.md、memory/YYYY-MM-DD.md 以及相关记忆载体中抽取待处理片段，并保留必要的上下文边界信息。OpenClaw 当前文档已经明确这些文件是记忆的核心承载体。
2. **敏感实体检测与语义分析**。通过规则引擎与语义模型识别显式敏感实体、隐含语义风险和特殊场景标志。
3. **综合评分计算**。结合 PII、语义、检索暴露度、场景范围和生命周期信息，计算每条片段的综合敏感度得分。
4. **隐私等级分配**。依据评分区间与业务规则，把片段分配到 L0-L3。具体阈值不宜在研究报告中写死为固定工业标准，更稳妥的表述是“以实验校准的经验阈值为初始配置，并在原型验证中迭代调整”。
5. **元数据与标签入索引**。将隐私等级、来源可信度、使用目的、场景域、生命周期和授权建议作为元数据与向量/索引对象一并写入存储层。OpenClaw 的内建记忆引擎已支持基于 SQLite 的索引与混合检索，这为“标签伴随索引写入”提供了现实基础。
6. **策略联动执行**。根据分级结果触发不同的存储、检索、同步和共享策略。例如，L1 可在授权域内正常检索，L2 默认启用更严格审计与用途约束，L3 则禁止普通向量召回，仅允许在更高强度隔离环境下受控使用。

### 4.1.7 分类结果到策略控制的映射

分级分类的真正价值，在于把内容识别结果转化为可执行的治理策略。基于 IDS usage control 的思想，本文建议将分类结果直接映射为 usage policy，而不仅是作为存储层的备注信息。IDS 明确指出，usage control 关注的是对数据资产“可做什么、不可做什么”的限制与义务，属于 access control 的扩展。

在本课题中，可形成如下映射关系：

- L0 允许用于低风险检索增强；
- L1 仅允许在授权域内读取，禁止跨域原文共享；
- L2 可在受限场景中读取或汇总，但不得自由导出或用于第三方画像；
- L3 原则上仅允许在受控沙箱中分析，不得导出原文、不得自由同步、不得暴露给普通插件。

国家数据局行动计划对使用控制、数据沙箱、履约存证和全程溯源的要求，也为这种“分类结果直连策略执行”的设计提供了制度依据。

进一步看，五维模型还能支持更细粒度的动态控制。例如，同为 L2 的记忆，在“本地任务连续性”用途下可以允许受控召回；但在“外部共享”用途下则应自动提升限制级别，只允许统计输出或完全禁止共享。又如，一条处于“待删除”生命周期的 L1 记忆，即使内容本身不算高敏，也应立即退出同步与共享链路。这样，分级分类就不再是静态内容管理动作，而成为后续访问控制、同步控制、审计和撤销管理的统一前提。

### 4.1.8 本节小结

本节从向量检索场景出发，构建了面向手机智能体记忆治理的隐私分级分类框架。其核心贡献有三点：一是把记忆对象从“原始文本”扩展为“内容 + 表示 + 元数据”的复合治理对象；二是通过五维分类模型和综合敏感度评分，把内容风险、检索风险和传播风险统一到同一判定框架；三是将分类结果直接映射到 usage policy，为后续的加密存储、访问控制、跨设备同步和受控共享提供了可执行入口。

同时需要强调，分类分级本身不是隐私保护的终点。RAG membership inference 等研究已经提示，即便数据库中只保留检索对象，攻击者仍可能推断某条样本是否存在。因而，本节提出的分级分类体系，其真正价值在于为第 4.2 节和第 4.3 节提供统一、前置、可审计的控制基础，而不是试图仅凭打标签解决所有隐私风险。

## 4.2 记忆片段的安全加密存储与细粒度访问控制

### 4.2.1 问题提出与设计原则

第 4.1 节已经给出了面向记忆对象的隐私分级分类框架，但分级结果本身并不会自动转化为安全性。对手机智能体而言，真正的风险并不只发生在“记忆是否写入磁盘”这一时刻，还发生在“记忆是否进入索引”“是否会被普通检索召回”“是否会跨设备同步”“是否被第三方主体再次加工或转发”等多个环节。OpenClaw 官方文档表明，其长期记忆和日常记忆以 Markdown 文件保存在工作区中，并通过内建索引机制支持全文检索、向量检索和混合检索；同时，工作区只是默认当前工作目录，并非天然强隔离边界，只有启用沙箱并配置工作区访问权限后，工具访问范围才会真正受到约束。

因此，本节研究的核心问题是：如何把第 4.1 节形成的隐私等级（L0—L3）、五维分类字段和策略接口字段，进一步落实为可执行的分层存储、分级加密、细粒度授权、用途控制、场景隔离和全过程审计机制。国家数据局《可信数据空间发展行动计划（2024—2028年）》明确将使用控制、数据沙箱、履约存证和全程溯源纳入可信数据空间的关键能力；国际数据空间（IDS）则明确指出，使用控制（Usage Control）是访问控制的扩展，重点关注“获准访问之后还允许对数据做什么”。这意味着，记忆治理不能停留在“谁能读”，还必须回答“读完后能不能同步、能不能导出、能不能训练、能不能转发”。

基于上述认识，本文提出六项设计原则：一是**本地优先**，即高敏记忆默认保留在用户可控边界内；二是**分层存储**，即内容、索引和治理信息分开管理；三是**分级加密**，即不同隐私等级采用不同强度和不同边界的保护；四是**前置授权**，即在检索前完成授权判定，而不是召回后再补救过滤；五是**用途受限**，即访问获准后仍受持续约束；六是**全过程留痕**，即访问、同步、共享、撤销和删除都必须留下审计证据。该原则体系与第 4.1 节的分类逻辑保持一致，并为第 4.3 节的跨设备同步控制提供前置条件。

### 4.2.2 分层存储架构与加密对象划分

为了使安全控制真正可落地，本文将记忆对象划分为五类：原始片段、摘要片段、向量表示、检索索引、治理元数据。原始片段是最接近用户输入和系统观察的内容，通常包含最高密度的敏感信息；摘要片段虽然经过压缩，但仍可能保留偏好、行为模式和关系结构；向量表示和检索索引虽然不以自然语言出现，但仍承载可推断语义，不能视为天然脱敏对象；治理元数据则包含隐私等级、来源可信度、使用目的、场景域、生命周期、授权建议和审计状态，是后续策略执行的控制基础。OpenClaw 当前以 MEMORY.md、memory/YYYY-MM-DD.md 和内建索引数据库分别承载内容与检索功能，这为“内容层—索引层—治理层”的分层架构提供了直接工程基础。

基于此，本文建议构建三级存储架构：

1. **内容存储层**：保存原始片段和摘要片段。
2. **检索存储层**：保存向量表示、倒排索引、检索辅助信息。
3. **治理存储层**：保存分类标签、场景域、用途限制、审计记录、撤销状态和策略版本。

三层分离的根本理由在于，它们面临的威胁与保护目标并不相同：内容层主要防止明文泄露，检索层主要防止越权召回和表示推断，治理层主要防止策略被绕过、篡改或失效。一旦把三者混在同一边界中管理，就容易形成“内容、索引、权限信息一起暴露”的单点失陷问题。

从保护边界看，本文主张高敏对象坚持本地优先。也就是说，L2 和 L3 等级的原始片段原则上优先保留在用户控制的本地边界内，只把最小必要的辅助信息暴露给后续检索或同步流程；L0 和部分 L1 对象则可以进入相对宽松的索引环境，以支持普通个性化服务。需要说明的是，“本地优先”并不等于“本地天然安全”，因为 OpenClaw 文档已经明确指出，工作区不是硬隔离边界，必须结合沙箱和工作区访问权限才能形成更强约束。

### 4.2.3 分级加密策略与密钥层次设计

在第 4.1 节的 L0—L3 四层隐私分级基础上，本文建议采用“等级决定强度，对象决定方式”的分级加密策略。该策略的核心不是把所有对象都做同一种加密，而是根据隐私等级、对象类型和执行边界决定加密方式。

- 对于 **L0 级记忆**，可采用最小必要保护，重点是完整性校验和版本控制，因为该类对象主要服务于一般性检索增强。
- 对于 **L1 级记忆**，建议采用常规强加密，并与用户身份、设备状态或本地解锁机制关联，使其能够在用户控制边界内安全使用。
- 对于 **L2 级记忆**，应进一步强化密钥隔离和访问前校验，不同场景域不应默认共享同一把数据保护密钥；同时，对于需要对外提供统计或聚合结果的 L2 对象，不应把“差分隐私（Differential Privacy）”理解为一种静态存储加密方式，而应将其用于导出统计结果或共享结果阶段，以降低聚合分析中的再识别风险。
- 对于 **L3 级记忆**，原则上不宜以普通明文长期存在于常规工作区中，而应优先在可信执行环境（Trusted Execution Environment，TEE）或数据沙箱中受控处理。NIST 关于机密计算和硬件保护的资料指出，可信执行环境可为受保护代码和数据提供隔离执行区域，用于防止包括操作系统、虚拟机监控层等高权限软件的访问；谷歌和微软的机密计算资料也都把“数据使用中保护”建立在硬件隔离的可信执行环境之上。

本文将记忆对象 m 的存储密文表示为：

$$C_m=加密_{K_{u,l,d}}(M_m,A_m)$$

其中，$M_m$表示记忆对象明文，$A_m$表示与该对象绑定的附加认证信息，包括隐私等级、场景域、用途限制、版本号等，$K_{u,l,d}$表示针对用户 u、隐私等级 l、场景域 d 派生出的保护密钥。这个表达的含义是：密钥不只与用户绑定，还应与隐私等级和场景域绑定，从而避免“同一用户、不同场景、同一把密钥”的过度授权问题。

在密钥层次上，本文建议采用三级结构：

1. **用户主密钥**：与用户身份解锁机制绑定，用于建立用户级根信任。
2. **分级场景密钥**：由用户主密钥派生，绑定隐私等级和场景域，实现 L0—L3 以及个人域、工作域、第三方域之间的密钥隔离。
3. **会话密钥**：为一次会话或一次受控处理临时生成，会话结束后销毁，适合短期活动记忆和临时计算结果保护。

其派生逻辑可抽象表示为：

$$K_{u,l,d}=派生函数(K_u,l||d||u)$$

其中，$K_u$为用户主密钥，l 为隐私等级，d 为场景域，v 为策略版本或上下文版本。这个公式的作用在于说明：即便同一用户在不同场景或不同等级下访问记忆，也不应默认共享同一保护密钥。

需要强调的是，向量表示和检索索引也必须纳入加密与隔离视野。对检索增强生成系统的研究已经表明，外部检索数据库可能面临成员推断风险，而表示层也可能成为隐私泄露入口。因此，L2 和 L3 记忆对象不应默认进入“跨端公共索引”，其向量表示、检索条目和相关日志应与原始片段一样被视作高价值对象对待。

### 4.2.4 细粒度访问控制模型设计

如果说加密解决的是“数据静态存在时如何被保护”，那么访问控制解决的是“谁在什么条件下、出于什么目的、可以对什么记忆对象执行什么动作”。本文建议采用基于角色的访问控制（Role-Based Access Control，RBAC）、基于属性的访问控制（Attribute-Based Access Control，ABAC） 与**使用控制（Usage Control）**相结合的三层模型。

其中，基于角色的访问控制适合定义较稳定的主体边界，例如“个人助理智能体”“办公助理智能体”“第三方接入主体”“审计主体”等。基于属性的访问控制适合表达更细粒度的动态条件。NIST SP 800-162 明确指出，基于属性的访问控制是一种逻辑访问控制方法，其授权是通过评估与主体、客体、请求操作以及环境条件相关的属性，并将这些属性与策略、规则或关系相匹配后得出的。

在本文场景中，核心属性可划分为三组：

- **主体属性**：用户身份、主体角色、设备标识、设备可信度、所属场景域。
- **客体属性**：记忆隐私等级、记忆对象类型、来源可信度、创建时间、来源场景。
- **环境属性**：访问时间、网络环境、设备安全状态、是否处于受控沙箱、是否需要进一步确认。

基于此，可将访问判定写为：

$$D_{访问}=f_{角色}(s,o,a) \land f_{属性}(A_s,A_o,A_e,a)$$

其中，s 表示主体，o 表示客体，a 表示请求动作，$A_s$、$A_o$、$A_e$ 分别表示主体属性、客体属性和环境属性。该公式表示：只有当角色边界允许，且属性规则匹配时，访问请求才进入下一阶段。

在此基础上，再引入使用控制，用于判定“访问获准后还允许如何处理”。IDS 文档明确指出，使用控制是访问控制的延伸，关注的是对数据资产“允许做什么，不允许做什么”的限制及义务，而不只是访问许可本身。

因此，本文进一步定义用途判定：

$$D_{用途}=g(p,r,t,x)$$

其中，p 表示使用目的，r 表示保留与传播限制，t 表示时间有效期，x 表示是否允许训练、导出、共享、同步等后续动作。于是，记忆对象的最终有效权限可以写为：

$$P_{有效}=D_{访问}\land D_{用途}\land D_{秘钥} \land D_{沙箱}$$

其中，$D_{\text{密钥}}$表示对应分级密钥是否可用，$D_{\text{沙箱}}$表示是否满足强制受控执行环境要求。这个公式体现了本文的一个核心思想：只有“访问许可 + 用途许可 + 密钥可用 + 执行环境合规”同时满足时，记忆对象才真正可用。

### 4.2.5 零信任约束与场景域隔离

在零信任（Zero Trust）视角下，NIST SP 800-207 强调，不应因为用户或资产位于内网、设备属于组织、或处于“本机环境”就默认给予隐式信任；认证和授权应在会话建立前独立完成，并且同时考虑主体和设备。

将这一原则应用于手机智能体记忆系统，可以得到三个直接要求。第一，跨设备记忆访问必须同时校验用户身份和设备状态，不能因为“同一个账号”就默认允许读取全部长期记忆。第二，不同场景域之间不得因为同属一个用户而自动继承权限；也就是说，个人域的高敏记忆不能因为用户本人同时拥有工作域身份，就自动被办公智能体读取。第三，外部连接器或第三方服务请求记忆时，必须携带明确用途、可审计策略和有效期限制，不能以“辅助服务”名义获得无限期、无限用途的读取权限。

为了把上述原则落到实现上，本文建议为每次请求计算一个设备—环境可信度分数：

$$T_{环境}=\alpha T_{设备}+ \beta T_{网络} + \gamma T_{会话} + \delta T_{场景}$$

其中，$T_{\text{设备}}$表示设备完整性和安全状态，$T_{\text{网络}}$表示当前网络可信度，$T_{\text{会话}}$ 表示当前认证会话强度，$T_{\text{场景}}$ 表示请求是否发生在允许的场景域内，$\alpha,\beta,\gamma,\delta$为权重。当 $T_{\text{环境}}$低于阈值 $\tau$ 时，即使主体角色本身具备访问资格，也应触发拒绝或进一步验证。这个表达式不是现成标准公式，而是把零信任思想转化为本课题中可实现的决策参数化形式。

### 4.2.6 与 OpenClaw 工作区隔离和沙箱机制的集成

OpenClaw 的沙箱文档提供了非常关键的控制点，即工作区访问权限（workspaceAccess）。根据官方文档，当该字段为 "none" 时，工具只能看到沙箱工作区；为 "ro" 时，可只读挂载智能体工作区；为 "rw" 时，才允许读写挂载原工作区。OpenClaw 还提供多智能体场景下的沙箱与工具配置能力。

基于这一机制，本文建议将细粒度访问控制嵌入记忆相关技能或工具调用路径中。也就是说，在任何涉及长期记忆、近端会话记忆或高敏索引访问的操作执行前，都应先触发策略决策点完成授权判定；只有当第 4.2.4 节定义的 $P_{\text{有效}}=1$ 时，才允许真正读取内容、索引或执行共享动作。对于 L2 和 L3 记忆对象，还应把“是否强制在受控沙箱中执行”作为硬条件，而不是一个可选优化项。

进一步地，本文建议把工作区访问权限与隐私等级映射起来：

- **L0**：在授权前提下可允许普通只读检索。
- **L1**：默认仅允许在本场景域中只读召回，不允许任意写回。
- **L2**：默认要求在只读工作区或独立沙箱中处理，禁止直接暴露给第三方普通工具。
- **L3**：原则上仅允许在数据沙箱、可信执行环境或更高强度隔离容器内受控使用，不进入普通工作区读写链路。

这种设计的本质，是把第 4.1 节中的“场景域”和“是否强制沙箱执行”两个策略接口字段，真正落实为运行时边界约束，而不是停留在元数据层。

### 4.2.7 审计、撤销与遗忘机制

记忆治理必须支持审计、撤销和遗忘，否则“可控共享”就无法成立。国家数据局行动计划明确要求构建履约行为存证和全程溯源能力；IDS 的使用控制框架同样强调策略执行后的证据价值。

本文建议系统至少保留四类日志：

1. **访问日志**：记录谁在何时、以何种场景域访问了何种等级的记忆对象。
2. **策略判定日志**：记录访问为何被允许、拒绝或要求进一步验证。
3. **同步日志**：记录哪些记忆对象进入了跨设备同步或统计导出流程。
4. **共享与撤销日志**：记录共享对象、用途限制、有效期、撤销时间和撤销结果。

为了支持时效性控制，本文建议把共享许可写为带有效期的约束函数：

$$V(t)= \begin{cases} 1, & t \in [t_0, t_{\text{到期}}] \\ 0, & \text{其他} \end{cases}$$

只有当 $V(t)=1$ 时，共享许可仍然有效；一旦超过有效期，系统应自动停止继续访问、继续同步或继续共享。

在遗忘机制方面，本文建议区分三类：

- **第一类是主动遗忘**，即用户明确提出删除请求时，同时删除内容存储层、检索存储层和治理存储层中的相关对象，并将该对象生命周期切换为“待删除/已删除”。
- **第二类是被动遗忘**，即对低使用频率、低价值或短期性记忆施加时间衰减，使其逐步退出活跃检索集。其权重可抽象为：

$$w_i(t) = w_i(0)e^{-\lambda t}$$

其中，$w_i(t)$为记忆对象 i 在时间 t 的有效权重，$\lambda$ 为衰减系数。该公式的含义是：随着时间推移而缺乏再次使用的记忆，其召回优先级应逐步下降，直至退出高频活跃集。

- **第三类是可验证遗忘**，即在删除或撤销后生成可供审计的删除凭证、撤销凭证或策略失效记录。这里需要说明，可验证遗忘在工程上属于更高要求的治理能力，报告中更适合写成“研究设计目标”或“增强方案”，而不宜写成当前系统已经完整具备的默认能力。

### 4.2.8 本节小结

本节在第 4.1 节“记忆隐私分级分类”的基础上，进一步构建了面向手机智能体记忆系统的安全存储与细粒度访问控制框架。其核心内容包括：将记忆对象拆分为原始片段、摘要片段、向量表示、检索索引和治理元数据，并建立内容层、检索层和治理层相分离的分层存储架构；依据 L0—L3 隐私等级实施差异化加密，并通过与用户、等级和场景域绑定的密钥层次降低跨场景过度授权风险；结合基于角色的访问控制、基于属性的访问控制和使用控制，实现“谁能访问、在何种条件下访问、访问后还能做什么”的连续治理；在零信任约束下，把用户身份、设备状态、网络环境和场景域共同纳入判定；依托 OpenClaw 的工作区访问权限和沙箱机制，将高敏记忆限制在受控执行边界内；最后，通过审计、撤销和遗忘机制，为记忆共享的可追溯性和可回收性提供支撑。

## 4.3 跨设备记忆同步的差分隐私保护

### 4.3.1 问题提出与同步边界

跨设备记忆同步是手机智能体形成长期连续体验的关键能力，但它同时也是隐私风险被放大的主要通道。OpenClaw 当前把长期记忆、日常记忆和可选的梦境摘要保存在工作区文件中，并支持通过检索机制跨会话调用这些记忆；一旦此类对象被直接复制到另一设备、云端或第三方环境，原本局限在单设备边界内的风险就会变成多边界传播风险。因此，本文不建议把“全量原始记忆跨设备同步”作为默认路径，而主张按照记忆对象类型、隐私等级、场景域和使用目的实行分层同步：低敏摘要和稳定偏好可在授权后同步，中敏记忆优先同步摘要、统计量或受限表示，高敏原文默认仅留在本地边界内。这个方向与 Gboard 的公开实践是一致的：其联邦学习（Federated Learning）做法并不是上传原始输入数据，而是在设备侧训练并仅上送受控的模型更新。

从第 4.1 节和第 4.2 节的统一术语出发，本节同步对象仍然限定为五类：原始片段、摘要片段、向量表示、检索索引、治理元数据；同步决策仍受五维分类字段约束，即内容敏感度、来源可信度、使用目的、场景域和生命周期。换句话说，跨设备同步不是一个独立模块，而是前两节“先分类、再加密、后授权”的自然延伸。只有当某个记忆对象在分类、加密、访问控制和场景隔离上都满足前置条件时，它才有资格进入同步流程。

基于上述认识，本文将本节目标定义为四项：第一，明确同步的隐私单位，是“用户级”还是“事件级”；第二，明确采用本地差分隐私（Local Differential Privacy）还是中心差分隐私（Central Differential Privacy），或两者混合；第三，建立与 L0—L3 隐私等级联动的异质化隐私预算分配机制；第四，在个性化收益、隐私损失、通信开销和端侧能耗之间形成可调节的平衡。NIST SP 800-226 明确指出，差分隐私不是一个单一数字承诺，而是一整套围绕威胁模型、数据收集暴露面、访问控制、侧信道和实现正确性的系统性评估框架；因此，本节将差分隐私视为“预算与风险管理体系”，而不是“额外加一层噪声”的局部技巧。

### 4.3.2 差分隐私基础与本课题适用设定

差分隐私（Differential Privacy）的标准形式可写为：若随机机制 M 对任意两个相邻数据集 $D$ 与 $D'$，以及任意输出事件集合 S，都满足

$$\Pr[M(D)\in S] \le e^{\varepsilon}\Pr[M(D')\in S] + \delta$$

则称 $M$ 满足 $(\varepsilon,\delta)$ 差分隐私。其中，$\varepsilon$越小，单个实体对结果可见性的影响越小；$\delta$ 则表示出现较大偏离的容忍概率。NIST SP 800-226 将差分隐私定义为一种量化实体在其数据出现在数据集时所承受隐私损失的数学框架；Rényi 差分隐私（Rényi Differential Privacy）则由 Mironov 提出，用于更紧地分析多轮机制组合后的隐私损失，Wang 等人进一步给出了与分析型矩会计（Analytical Moments Accountant）相关的组合分析结果。

对手机智能体跨设备同步而言，至少存在两种自然的隐私单位。其一是事件级差分隐私，即把一条记忆片段、一条摘要或一次局部更新视为一个隐私单位；其二是用户级差分隐私，即把某个用户在一轮或一段时间窗口内贡献的全部同步信息视为一个整体隐私单位。Gboard 的工业实践更接近用户级保护语境，因为它讨论的是用户设备参与训练与更新时的形式化保证；对手机智能体而言，若同步对象包含稳定偏好、长期习惯和持续决策模式，用户级差分隐私通常比事件级差分隐私更符合真实风险边界，但代价也更高。基于此，本文建议：对于 L0—L1 的低敏同步对象，可优先按事件级预算管理；对于 L2 的中高敏同步对象，应尽可能向用户级预算靠拢；对于 L3 的核心敏感对象，原则上不进入普通云辅助同步。

在机制选择上，本地差分隐私强调在设备端加噪后再上传，服务端只看到已扰动的数据；中心差分隐私则通常依赖一个受控聚合方，在聚合或训练后提供整体隐私保证。对手机智能体而言，两者并非互斥关系。若同步对象是简单统计量、偏好分布或轻量标签，本地差分隐私更适合，因为敏感信息在离开设备前已经得到保护；若同步对象是模型更新、摘要向量或聚合参数，则可以通过“本地裁剪 + 安全聚合 + 聚合后差分隐私会计”的混合路径兼顾效用与保护。本文据此采用“本地优先、混合实现”的设定，而不把所有同步都强行归为同一种差分隐私机制。这个判断与 Gboard 的联邦学习实践、NIST 对差分隐私系统性评估的要求以及安全聚合文献中的常见部署方式是一致的。

### 4.3.3 分层同步对象与两类同步模式

结合前两节的分层治理框架，本文建议把跨设备同步对象分为四类：第一类是低敏摘要与稳定偏好，如通用偏好标签、非敏感任务模板、常规提醒规则；第二类是中敏摘要与受限表示，如压缩后的近期上下文、去标识化后的行为摘要、经裁剪后的向量更新；第三类是局部统计量与模型更新，如频次统计、偏好分布、排序参数、联邦优化更新；第四类是高敏原文与高敏关联表示，如核心身份线索、财务健康信息、强可重识别的长期决策模式。本文主张前三类按等级和场景受控同步，第四类原则上不走普通云辅助同步通道。

在同步模式上，本文设计两种路线。第一种是**本地优先同步**，即设备之间通过点对点加密链路、局域网受控通道或同一用户信任边界下的直接同步完成数据迁移，适用于高敏对象、短距离同步或同一用户显式确认的场景。这一路线更强调“数据留在用户控制边界内”，适合 L3 对象和部分 L2 对象。第二种是**云辅助同步**，即各设备仅上传经裁剪、加噪和策略过滤后的更新或统计信息，由聚合方完成安全聚合和受控下发，适用于 L0—L2 中可共享的同步对象。这里的“云辅助”并不意味着服务端可见单设备明文数据；相反，其核心前提是服务端只应看到聚合结果或受限表示。安全聚合文献将其定义为多方在不泄露各自私有输入的前提下计算总和之类聚合值的方法，谷歌的安全聚合论文就是针对“用户手持设备上的高维更新向量”提出的。

为了把同步资格形式化，本文为每个记忆对象 $m_i$ 定义同步可行性函数：

$$\chi_i = \mathbf{1}\bigl[\ell_i \le \theta_d \land p_i \in \mathcal{P}_d \land s_i \in \mathcal{S}_d \land v_i = 1 \bigr]$$

其中，$\ell_i$表示记忆对象的隐私等级，$\theta_d$表示在场景域 d 下允许进入同步流程的最高等级阈值，$p_i$表示该对象的使用目的，$\mathcal{P}_d$表示该场景域允许的用途集合，$s_i$表示对象类型，$\mathcal{S}_d$ 表示该场景域允许同步的对象类型集合，$v_i$表示其生命周期和撤销状态是否有效。只有当 $\chi_i=1$时，该对象才允许被编码为同步输入。这个公式的意义是：同步资格首先由前两节的分类与策略结果决定，而不是由“用户有多台设备”这一事实自动触发。

### 4.3.4 核心同步协议设计

本文建议的云辅助同步协议采用“三阶段”结构：设备侧本地处理、聚合侧安全聚合、结果侧受控下发。这一结构与 Gboard 的联邦学习部署逻辑、安全聚合文献中的常见协议形态，以及 NIST 对差分隐私系统性评估的要求相一致。

**第一阶段是设备侧本地处理**。设备先根据第 4.1 节和第 4.2 节的分类、加密和访问控制结果，选出满足 $\chi_i=1$ 的同步对象；随后，将可同步对象转化为同步表示 $u_i$，该表示可以是摘要向量、偏好统计、局部模型更新或其他受控参数。为了限制单设备贡献的敏感度，设备侧需先执行范数裁剪，再注入差分隐私噪声。对第 i 个设备，一种典型形式可写为：

$${u}_i = \operatorname{clip}(u_i, C_i) + \mathcal{N}(0,\sigma_i^2 C_i^2 I)$$

其中，$\operatorname{clip}(u_i, C_i)$ 表示把更新裁剪到阈值 $C_i$内，$\sigma_i$表示噪声强度，I 为单位矩阵。这个表达式体现的是“先裁剪、后加噪”的标准差分隐私更新思路。Gboard 在 DP-FTRL 路线中利用类似的“受控更新 + 差分隐私会计”思想实现形式化保证。

**第二阶段是聚合侧安全聚合**。设备不直接上传可被单独解释的原始更新，而是通过安全聚合协议只让聚合方看到总和或平均值。设参与本轮同步的设备集合为 $\mathcal{U}_t$，则服务端看到的应是：

$$A_t = \sum_{i \in \mathcal{U}_t} \tilde{u}_i$$

而不是单个 $\tilde{u}_i$的明文值。Bonawitz 等人的安全聚合工作就是围绕这一目标展开：保护各设备私有更新，仅暴露聚合结果。后续关于高效差分隐私安全聚合的工作也说明，在分布式联邦场景中，可以把差分隐私与安全聚合结合起来，避免服务端观察到单设备贡献。

**第三阶段是结果侧受控下发**。服务端或协调节点根据聚合结果生成可下发的同步状态 $G_t$，它可以是更新后的偏好分布、排序参数、摘要模板或轻量模型权重，但不得把高敏原文直接重新广播到其他设备。若采用中心差分隐私校正，还可在聚合后对结果进一步加噪。结果下发前仍需经过第 4.2 节定义的用途控制和场景域校验，只有在目标设备的授权、密钥、环境可信度和同步目的均满足条件时，才允许真正写入本地记忆或本地模型。用统一形式表达，可写为：

$$G_t = h(A_t,\ \Pi_t)$$

其中，$h$ 表示聚合后生成同步结果的函数，$\Pi_t$表示本轮生效的同步策略，包括可写入对象类型、场景域限制、用途范围和有效期。这个表达式强调：跨设备同步的输出不是原样复制，而是“经过策略裁剪后的受控结果”。

### 4.3.5 隐私预算管理与异质化分配

差分隐私在本课题中不应被理解为“全系统统一一个 ε”的静态配置，而应被设计为与记忆分类体系联动的异质化预算系统。NIST SP 800-226 强调，评估差分隐私保证时不能只看最终 $(\varepsilon,\delta)$ 数值，还必须考察威胁模型、数据收集暴露面、访问控制与实现方式；这意味着预算管理必须嵌入整个同步流程，而不是只在算法层孤立存在。

本文建议为每类记忆对象分别分配预算上限。设总预算为 $\varepsilon_{\text{总}}$，按隐私等级和对象类型分成若干桶，则可写为：

$$\varepsilon_{\text{总}} = \varepsilon_{L0} + \varepsilon_{L1} + \varepsilon_{L2} + \varepsilon_{L3}$$

其中，通常满足 $\varepsilon_{L0} \ge \varepsilon_{L1} \ge \varepsilon_{L2}$，而 $\varepsilon_{L3}$ 在默认云辅助同步路径中接近于 0，因为 L3 对象原则上不进入普通同步。进一步地，对同一等级内部，还可以按对象类型再划分，例如偏好统计、摘要更新和模型更新分别拥有不同预算。这样做的目的，是把“越敏感、越少同步、预算越紧”变成系统内在规则，而不是依赖人工临时判断。

在多轮同步场景中，预算会累积消耗，因此需要采用会计机制进行追踪。Rényi 差分隐私与分析型矩会计常用于更紧地处理多轮组合损失。若设第 t 轮同步在某一 Rényi 阶数 $\alpha$ 下的损失为 $\rho_t(\alpha)$，则总损失可表示为：

$$\rho_{\text{总}}(\alpha) = \sum_{t=1}^{T} \rho_t(\alpha)$$

随后再将其换算为最终的 $(\varepsilon,\delta)$保证。Mironov 的 Rényi 差分隐私工作与 Wang 等人的分析型矩会计工作都为这种多轮组合提供了理论基础。对本课题而言，这一机制尤其重要，因为跨设备记忆同步往往是长期持续发生的，若不做会计追踪，单轮看似温和的预算也可能在长期中失控。

基于此，本文建议采用异质化分配策略：L0 级低敏偏好或一般模板可使用相对宽松预算并允许较高同步频率；L1 对象采用中等预算与周期性同步；L2 对象采用较严格预算、低频同步和更强裁剪；L3 对象默认不走云辅助同步，只在本地优先模式下经用户确认和强边界保护执行有限同步。这一分配方式既与第 4.1 节的分级逻辑一致，也与 Gboard 等工业实践“保护强度和效用权衡”的思路一致。

### 4.3.6 个性化、隐私与能耗的权衡

差分隐私保护不会“免费”出现。加噪、裁剪、安全聚合和多轮会计都会带来个性化精度下降、通信成本上升和端侧能耗增加的问题。Gboard 论文强调，其工业部署之所以需要 carefully choose client participation criterion，本质上就是在效用与隐私之间做工程平衡；NIST SP 800-226 也强调，对差分隐私系统的评估必须同时考虑效用偏差、正确性和周边系统因素。

为此，本文建议把同步优化目标写成一个多目标问题：

**个性化—隐私—资源三目标优化**

$$\max_{\mathcal{Q},\,\varepsilon,\,f} \; U(\mathcal{Q},f) - \lambda_1 L_{\text{隐私}}(\varepsilon) - \lambda_2 C_{\text{通信}}(f) - \lambda_3 E_{\text{能耗}}(f)$$

其中，$\mathcal{Q}$表示进入同步的对象集合，$\varepsilon$表示预算配置，$f$ 表示同步频率，$U$表示个性化效用，$L_{\text{隐私}}$表示隐私损失，$C_{\text{通信}}$表示通信成本，$E_{\text{能耗}}$表示端侧能耗，$\lambda_1,\lambda_2,\lambda_3$为权重。这个公式不是标准教科书结论，而是把本课题的工程目标形式化：同步方案不应只优化精度，也不应只压低 $\varepsilon$，而应在“个性化—隐私—资源”三者之间找平衡点。

在更前沿的方向上，ALPINE 这类 2025 年预印本研究尝试根据动态风险、资源约束和任务需求实时调节差分隐私级别，用强化学习实现“隐私—效用—能耗”的动态平衡。对本课题而言，它更适合作为“未来可引入的自适应预算控制思路”，而不宜作为当前成熟工业基线。换言之，本课题现阶段更稳妥的做法仍是：先采用分级预算、策略约束和会计追踪建立可解释的基线机制，再视实验结果决定是否引入自适应调节。

### 4.3.7 用户可控与策略联动

为了让跨设备同步真正符合“个人记忆可控共享”的目标，本文建议把差分隐私预算与用户控制界面和策略系统打通。也就是说，用户不必直接面对复杂的数学细节，但系统至少应提供两层控制：第一层是同步模式选择，例如“严格保护模式”“均衡模式”“高性能模式”；第二层是场景域与对象级开关，即用户可分别控制个人域、工作域、第三方域中哪些对象允许进入同步。其底层对应的是不同的 $\varepsilon$ 配置、同步频率和对象白名单，而上层呈现给用户的则是“更保守”或“更连续”的体验模式。NIST 对差分隐私评估的强调之一，就是不同承诺必须和实际威胁模型与系统行为对齐；因此，用户可控不只是产品体验需求，也是防止“数学承诺与实际使用脱节”的重要机制。

从与前两节的衔接看，本节建议继续沿用五个统一策略接口字段：是否允许索引、是否允许普通召回、是否允许跨设备同步、是否允许外部共享、是否强制沙箱执行。其中，“是否允许跨设备同步”由本节给出更具体的执行规则，而其余字段依然与第 4.1、4.2 节共同决定同步前提。例如，一条 L2 记忆即使从差分隐私角度可被低频同步，如果第 4.2 节的用途控制明确禁止其用于跨设备共享，或其生命周期已进入待删除状态，则该对象依然不得进入同步队列。由此，差分隐私不再是一个独立附加层，而是嵌入整个记忆治理流程中的一环。

### 4.3.8 本节小结

本节围绕“跨设备记忆同步如何在长期连续体验与隐私保护之间建立平衡”展开，形成了与前两节统一的技术框架：首先，明确同步对象不应默认等于原始记忆，而应按隐私等级和对象类型分层处理；其次，采用“本地优先同步 + 云辅助同步”双模式架构，使高敏对象尽量留在用户控制边界内，中低敏对象通过联邦式更新、安全聚合和差分隐私实现受控同步；再次，通过“先裁剪、后加噪、再聚合、后下发”的协议流程，把单设备更新转化为受限的聚合结果；最后，通过 Rényi 差分隐私会计和异质化预算分配，把隐私损失管理与 L0—L3 等级体系联动起来，并将个性化效用、通信和能耗纳入统一权衡。Gboard 的公开工业实践、NIST 的差分隐私评估指南、Rényi 差分隐私与分析型矩会计研究，以及安全聚合文献，共同支撑了这一设计。

## 4.4 OpenClaw 工作区隔离与可信数据空间数据沙箱融合

### 4.4.1 问题提出与融合目标

到第 4.3 节为止，本文已经分别建立了记忆对象的分级分类、分层存储与加密访问控制、以及跨设备同步的差分隐私保护机制。但这些机制要真正落地，还必须回答一个更底层的问题：记忆在哪个执行边界内被读取、处理、聚合和导出。如果执行边界本身不清晰，那么即使完成了分类、加密和同步预算配置，高敏记忆也仍可能在错误的运行环境中暴露。OpenClaw 官方文档已经明确区分了工作区（workspace）、智能体（agent）、沙箱（sandbox）和网关（gateway）的不同角色：工作区更接近逻辑上的持久化边界，沙箱才是运行时访问边界，而网关承载控制平面和策略平面；其安全文档进一步强调，一个网关默认对应一个“受信操作者边界”，并不适合多个相互不信任或对抗性操作者共享使用，混合信任场景应拆分信任边界。

与此同时，可信数据空间的相关正式文件则把“数据沙箱”明确界定为一种可信数据管控技术：通过构建应用层隔离环境，允许数据提供方、使用方和服务方在安全隔离区域内对数据进行使用、分析、处理、计算和结果输出；并要求以“环境安全、执行可信、行为可证”为目标，结合使用控制、数字合约、行为校验、操作执行和行为记录等能力实现数据流通利用过程“符合预期”。这意味着，可信数据空间并不鼓励数据在不同主体和不同系统间自由拷贝，而更强调在受控环境中完成受限使用。

基于这两类边界模型，本文提出“双层边界融合”目标：第一层以 OpenClaw 的智能体、工作区和沙箱建立应用执行边界；第二层以可信数据空间的数据沙箱、使用控制和数字合约建立数据使用边界。前者解决“谁在什么运行环境中可以接触哪些工具和文件”，后者解决“即使允许接触，数据还能被怎样处理和输出”。二者融合后，系统不再以“把整条记忆发给另一个环境”为默认思路，而是转向“把必要能力带到受控环境中，对高敏记忆执行受限处理，并仅输出最小必要结果”。

### 4.4.2 OpenClaw 隔离语义与信任边界澄清

为了避免把“逻辑分区”误当成“安全隔离”，有必要先澄清 OpenClaw 当前隔离相关概念的语义。其配置文档表明，默认工作区路径为 ~/.openclaw/workspace；沙箱配置则可选择不同后端，并提供 workspaceAccess 选项，支持 "none"、"ro"、"rw" 三种工作区访问方式。其默认 Docker 沙箱配置还显示，沙箱可设置为只读根文件系统、使用临时文件系统、默认关闭网络、丢弃全部 Linux 能力等。这说明 OpenClaw 已提供较明确的运行时收缩手段，但这些手段并不等于“任何多智能体/多场景部署都天然隔离”。

OpenClaw 官方安全文档对此表述得非常直接：如果有人能修改网关所在主机的状态或 ~/.openclaw 配置，就应被视为受信操作者；在一个网关中，经过身份认证的操作者是“控制平面角色”，而不是“每用户独立租户角色”；会话标识符和会话标签只是路由选择器，而不是授权令牌；如果多个不同的人都能向同一个带工具的智能体发消息，他们共享的其实是同一套权限，而不是各自独立的宿主授权边界。更进一步，官方明确写道：多个相互不信任或对抗性的操作者共享同一个网关并不是推荐部署方式，混合信任团队应拆分网关，至少拆分操作系统用户或主机。

这种边界澄清对本课题非常关键，因为它说明：个人域、工作域和第三方域的记忆隔离，不能只依赖“不同智能体名称”或“不同会话标识”来实现。前述社区 issue 也从另一个侧面反映了这一点：多个智能体若共享同一记忆数据库或同一命名空间，可能发生“记忆串扰（memory bleed）”，导致一个智能体召回另一个智能体的记忆。这类问题虽不等同于官方架构缺陷定论，但足以说明“逻辑分组”与“真正隔离”之间仍存在差距。

由此，本文将 OpenClaw 侧的执行边界分为三类：**逻辑边界**、**运行边界** 和 **信任边界**。逻辑边界由智能体标识、工作区目录和会话路由构成；运行边界由沙箱后端、工作区访问权限、网络和文件系统策略构成；信任边界则由“是否独立网关、独立操作系统用户、独立宿主”决定。只有当三者同时被清晰定义时，工作区隔离才有现实安全意义。

### 4.4.3 可信数据空间数据沙箱的融合思路

可信数据空间技术文件把数据沙箱定义为“应用层隔离环境”，其目标不是简单“把数据放进去”，而是让数据提供方、使用方和服务方在安全隔离区域内完成使用、分析、处理、计算和结果输出；同一文件还指出，接入连接器作为数据实际使用的执行点，应具备策略解析、行为校验、环境构建、操作执行和行为记录存证等能力；当连接器自身能力不足，可信数据空间服务平台还可提供集中式使用控制环境，确保数据访问、分析、计算和销毁等行为可控。

基于这一思路，本文提出的融合方案不是“把 OpenClaw 整体搬进数据空间”，而是将其拆分为两层：

- **第一层是 OpenClaw 执行层**，负责会话编排、工具调用、工作区读写和本地记忆治理；
- **第二层是 可信数据空间受控使用层**，负责高敏共享记忆的分析、聚合、统计和受限结果生成。

在此结构下，高敏记忆对象不再直接下发到普通工作区，而是先由使用控制策略和数字合约决定是否可进入数据沙箱，再由数据沙箱完成受控计算，最终输出统计结果、最小必要片段或受限数据产品，而不是原始整条记忆。国家数据局技术架构文件明确指出，可信数据空间的目标是通过数字合约描述特定参与方对数据内容、使用方式、使用次数等的预期，并通过集成在软硬件环境中的使用控制技术，对使用数据的算法、应用进行控制。本文的融合方案正是把这一要求具体化到手机智能体记忆场景中。

为了把“双层边界”形式化，本文定义跨域记忆处理的有效执行条件为：

$$P_{\text{跨域有效}} = D_{\text{执行边界}} \land D_{\text{数据沙箱}} \land D_{\text{使用控制}} \land D_{\text{数字合约}} \land D_{\text{审计}}$$

其中，$D_{\text{执行边界}}$ 表示 OpenClaw 侧的工作区、沙箱和信任边界配置满足要求；$D_{\text{数据沙箱}}$表示请求在合规的数据沙箱环境中执行；$D_{\text{使用控制}}$ 表示用途、场景域、次数、时间等限制满足策略；$D_{\text{数字合约}}$表示当前跨域处理符合已绑定的数字合约；$D_{\text{审计}}$表示行为记录与存证链路可用。只有当上述条件同时满足时，高敏记忆才允许跨域进入受控处理流程。这个公式的核心含义是：跨域处理不是单一授权动作，而是多重边界共同成立的结果。

### 4.4.4 场景域隔离与最小必要输出

结合前文定义的场景域，本文建议至少区分三类运行区域：个人域、工作域、第三方域。个人域主要承载用户私人偏好、长期生活习惯和高敏个人记忆；工作域主要承载项目资料、会议上下文、组织流程和办公任务；第三方域主要承载外部服务、插件、平台连接器或可信数据空间中的其他参与方调用。OpenClaw 官方已经说明，同一网关内的多个会话和多个智能体并不自动构成多租户授权隔离，因此，个人域和工作域若要实现真实隔离，至少应分别部署在不同的工作区、沙箱范围，必要时部署在不同的网关、操作系统用户甚至宿主边界之中。

在这种场景划分下，跨域共享不应以“整条记忆直接流动”为默认方式，而应以“最小必要输出”为原则。国家数据空间技术架构文件明确指出，可信数据空间中的基本流通单元是数据产品，数据产品可以是数据库表、数据文件或数据接口，而不必等于原始数据资源本身。由此，本文建议把高敏记忆跨域处理后的输出限定为三类：最小必要片段、统计结果、受限数据产品。例如，第三方域若需要利用用户习惯，只能获得“聚合后的偏好向量”或“受限评分结果”；工作域若需要个人日程相关信息，只能获得“是否存在冲突”或“可用时间窗”，而不是完整私人记忆文本。

本文用下式刻画跨域最小必要输出：

$$y = \operatorname{最小必要输出}(m,\ \Pi,\ \mathcal{F})$$

其中，$m$ 表示输入记忆对象，$\Pi$ 表示绑定的使用控制策略和数字合约，$\mathcal{F}$ 表示允许在数据沙箱内执行的处理函数集合。输出 $y$ 不再等同于原始记忆，而是经过策略裁剪和功能约束后的结果。若进一步细分，可把 $y$ 限定为：

$$y \in \{\text{最小必要片段},\ \text{统计结果},\ \text{受限数据产品}\}$$

这个表达式的意义在于把“数据不自由流动，而是能力在受控环境中流动”的设计原则形式化：外部主体需要的不是原始记忆本身，而是在受控环境中由原始记忆产生的、最小必要且符合用途约束的结果。

### 4.4.5 运行时策略执行与行为校验

可信数据空间使用控制技术文件提出了一个很适合本课题吸收的链路：策略解析—行为校验—操作执行—行为记录与存证。OpenClaw 侧也提供了合适的接入点，因为其记忆读取、工具调用和沙箱执行本身就是显式的运行时动作，而不是隐含在模型内部的不可观测过程。

基于此，本文建议把与高敏记忆有关的任何跨域操作统一抽象为请求 $r$，其中包含主体、客体、动作、场景域、用途、环境状态和合约标识等字段。运行时先进行行为校验，再决定是否执行。可形式化为：

$$D_{\text{行为}} = \phi(A_s, A_o, A_e, p, d, c)$$

其中，$A_s$表示主体属性，$A_o$ 表示客体属性，$A_e$ 表示环境属性，$p$ 表示使用目的，$d$ 表示场景域，$c$ 表示数字合约。若 $D_{\text{行为}}=1$，则允许进入操作执行；若为 0，则拒绝；若为中间状态，则触发进一步验证或人工确认。这与第 4.2 节的访问控制模型天然兼容，只是这里把“环境是否位于数据沙箱中”“是否符合数字合约”也纳入了统一判定。

在 OpenClaw 具体实现上，一个现实可行的接入点是：在任何会导致跨域记忆读取、跨域工具调用或高敏结果导出的技能执行前，先调用策略决策点完成授权验证；只有当 $P_{\text{跨域有效}}=1$ 且 $D_{\text{行为}}=1$ 时，才允许继续执行。对于高风险操作，还应当引入人工在环（Human-in-the-loop）确认，即任务先挂起，再由用户或管理员确认是否继续。OpenClaw 的安全文档本身也强调，共享同一工具权限集的主体会共同影响同一智能体行为，因此在高风险场景中引入人工确认是一种必要的风险补偿，而不是可有可无的体验选项。

### 4.4.6 隔离强度分层与部署路线

考虑到个人域、工作域和第三方域的敏感度不同，本文不建议把所有场景都部署在同一种隔离强度之下，而建议采用分层隔离路线。最低一层是进程级隔离，适合低敏任务和一般性工作流，把不同场景的智能体运行在不同进程中，以降低误共享状态的概率；中间一层是容器级隔离，结合 OpenClaw 自带沙箱或类似 OpenShell 的运行时，在内核级或容器级边界上限制本地文件、凭证和外部网络访问；最高一层是可信执行环境级隔离，适合 L3 级核心敏感记忆，在受保护的硬件飞地中执行高敏处理。OpenClaw 官方配置已经支持 Docker 和 OpenShell 等沙箱后端；NVIDIA OpenShell 官方文档将其定义为面向自主智能体的开源运行时，提供内核级隔离和声明式策略；其 GitHub 文档还显示，每个沙箱默认从最小出站访问开始，额外的网络能力需要通过简短的策略显式开放。

在更高隔离场景下，NEAR AI 官方页面已把“在可信执行环境中运行 OpenClaw”作为明确方案来介绍，称其可让代码和数据在安全飞地中运行，连操作系统和云服务提供方也无法检查其中内容；页面还明确指出，长期记忆、凭证和工具访问可以留在加密内存中。NVIDIA NemoClaw 的官方页面则把自己描述为给 OpenClaw 增加隐私与安全控制的开源堆栈，并说明其通过 OpenShell 为 OpenClaw 增加基于策略的隐私和安全护栏。由此可见，围绕 OpenClaw 的工程生态已经出现了“容器级加强”和“可信执行环境级加强”两种现实路线，可以作为本课题在不同敏感等级场景中的参考实现。

为了把隔离强度与场景域、隐私等级统一起来，本文定义隔离层级选择函数：

$$I^\ast = \arg\min_{I \in \{\text{进程级},\text{容器级},\text{可信执行环境级}\}} \Bigl\{ \operatorname{风险}(m,d,p) \le \operatorname{能力上界}(I) \Bigr\}$$

其中，$m$ 表示记忆对象集合，$d$表示场景域，$p$ 表示使用目的。该公式表示：对某一类记忆和某一类用途，应选择满足风险要求的最低足够隔离层级，而不是盲目把所有处理都推到最高成本环境中。这样既能控制部署复杂度，又能保持高敏对象的保护强度。

### 4.4.7 审计、解释与跨域串扰缓解

双层边界融合带来的另一个直接好处，是能够显著提升审计清晰度和合规解释能力。国家数据局使用控制技术要求明确提出，接入连接器应支持细粒度日志记录、查询和审计追溯，并按要求向平台回传存证；行动计划也要求构建空间合约和履约行为存证体系，提升全程溯源能力。

在本文框架下，任何一次高敏记忆跨域处理都至少应留下五类证据：边界证据、策略证据、合约证据、执行证据、输出证据。边界证据说明请求在哪个 OpenClaw 工作区、哪个沙箱后端、哪个数据沙箱环境中执行；策略证据说明使用控制规则如何生效；合约证据说明本次处理依据哪个数字合约；执行证据记录请求、校验和操作过程；输出证据记录最终输出的是最小必要片段、统计结果还是受限数据产品。这样，“为什么某条记忆没有被直接共享，而是只输出了统计结果”就能得到可审计、可解释的回答。

这种设计还能够缓解多智能体记忆串扰问题。社区关于记忆数据库共享导致 memory bleed 的 issue 已经表明，仅仅依赖“不同智能体名”或“不同会话标签”并不能充分让用户信任。若把个人域与工作域分别部署在独立信任边界中，并把跨域处理统一收口到数据沙箱和使用控制链路，就能把“逻辑串扰”转化为“显式跨域请求”，进而纳入审计和合约管理。

### 4.4.8 面向可信数据空间的记忆共享协议映射

在第 4.4 节前文中，本文已经提出了“OpenClaw 应用执行边界 + 可信数据空间数据使用边界”的双层边界融合思路。为了进一步增强该思路的协议化表达能力，有必要引入面向多智能体记忆共享的前沿研究作为参考。当前较有代表性的两项工作分别是 SAMEP（安全智能体记忆交换协议） 和 Collaborative Memory（协作记忆框架）。其中，SAMEP 由 2025 年 arXiv 预印本提出，目标是为智能体之间建立一种“持久化、安全、可语义检索”的记忆交换协议，并强调细粒度访问控制、分布式记忆仓和与 MCP、A2A 等通信协议兼容的接口设计；Collaborative Memory 同样是 2025 年的 arXiv 研究，重点解决多用户、多智能体环境下的非对称、时变访问控制问题，并提出 private memory 与 shared memory 两层结构，以及基于来源、资源和时间戳的可追溯属性。需要明确的是，这两项工作目前更适合作为研究性参考框架，而不宜被表述为已经广泛落地的工业标准。

从结构上看，SAMEP 关注的是“记忆如何交换”，Collaborative Memory 关注的是“记忆交换后如何受限可控”。这两项研究与前文提出的可信数据空间共享机制具有较强互补性。SAMEP 所强调的持久化上下文共享、语义检索、细粒度访问控制和审计追踪，与本文第 4.1 节的分级分类、第 4.2 节的细粒度访问控制以及第 4.4 节的数据沙箱受控共享是高度兼容的；Collaborative Memory 所强调的 private/shared 双层记忆、非对称时变权限和基于来源的追溯机制，则能够进一步补强本文已提出的“个人域—工作域—第三方域”场景域隔离模型。也就是说，前者更适合映射为跨智能体记忆交换接口层，后者更适合映射为跨主体共享的权限与审计约束层。

基于上述理解，本文建议将面向可信数据空间的记忆共享协议映射为五个逻辑层次。

**第一层是身份与连接层**，由可信数据空间连接器负责主体接入、统一身份认证和参与方标识管理。FIWARE 数据空间连接器的公开资料表明，连接器已经演化为一组部署在受控环境中的组件集合，其定位正是把组织安全接入到数据空间中，并支撑与 Gaia-X、IDS 等框架兼容的数据主权要求；TANGO 白皮书也明确说明其采用了 FIWARE 数据空间连接器，并以此承接数据主权和合规要求。

**第二层是记忆交换层**，可参考 SAMEP 的接口思想，把跨智能体共享对象从“原始整条记忆”收缩为“带有治理元数据的受控记忆对象”或“受限记忆视图”。在本文框架中，这意味着交换对象至少应附带隐私等级、来源可信度、使用目的、场景域、生命周期和授权建议，而不是只传递裸内容。这样做的原因在于，可信数据空间中的共享不是单纯把数据从一方复制到另一方，而是要让接收方在后续处理过程中继续受数据主权和使用控制约束。IDS 参考架构明确强调，数据主权要求数据使用控制权始终保留在数据收集、存储和提供者手中，而不应在共享时完全转移给外部平台或调用方。

**第三层是执行环境层**，由 OpenClaw 沙箱、容器级隔离或可信执行环境共同承接。前文已经指出，OpenClaw 工作区本身只是逻辑边界，不是强隔离边界；因此，一旦发生跨智能体或跨场景记忆调用，请求不应直接在普通工作区中执行，而应优先进入受控执行环境。若共享对象属于高敏等级，则进一步进入可信数据空间的数据沙箱或可信执行环境中处理，再返回最小必要结果。该层可以理解为对 SAMEP 所称“安全交换”与 Collaborative Memory 所称“权限约束视图”的工程化补充：即不仅交换接口受限，执行环境也必须受限。OpenClaw 官方安全文档关于 mixed-trust 场景应拆分信任边界的建议，正支撑了这一部署选择。

**第四层是使用控制与数字合约层**。这里可直接吸收可信数据空间技术文件和 IDS usage control 的思想，把“可否访问”扩展为“允许如何使用、允许使用多久、允许输出什么形式、是否允许再次传播”。TANGO 的第二份白皮书公开说明，其方案结合了区块链、数据空间连接器、自主主权身份和自加密/解密等技术，以增强数据共享过程中的信任与韧性；与此同时，IDS 公开材料明确强调，usage control 是维护 data sovereignty 的核心技术手段，用于在共享后持续约束数据的使用与转发方式。对本课题而言，这意味着 SAMEP 或 Collaborative Memory 若要真正映射到可信数据空间中，就不能只停留在“协议字段层面”，而必须与数字合约和使用控制规则联动。

**第五层是审计与可追溯层**。Collaborative Memory 强调 memory fragments 应携带 immutable provenance attributes，以支持回溯性权限检查；SAMEP 也把审计追踪作为其安全交换的重要一环。这与本文前文提出的“访问日志、同步日志、共享与撤销日志、合约与履约存证”高度一致。在可信数据空间环境下，这一层可以进一步与数字合约履约记录和连接器行为存证结合，使共享过程具备“谁授权、谁访问、何时访问、用于什么目的、是否越界输出”的完整证据链。换言之，SAMEP 和 Collaborative Memory 为本课题提供的价值，不在于替代前文的总体架构，而在于为“双层边界内的受控记忆共享”补充了更接近协议级与权限级的研究参考。

综合来看，本文建议将 SAMEP 主要作为“跨智能体记忆交换接口参考”，将 Collaborative Memory 主要作为“多用户、多场景、时变权限共享参考”，并通过可信数据空间连接器、使用控制、数字合约和数据沙箱把两者映射到受控共享环境中。这样，前文提出的“数据不自由流动，而是在受控边界内生成可共享的最小必要结果”就能获得更清晰的协议层解释，同时也为后续若单独讨论“记忆数据资产化与合规共享”提供接口基础。

### 4.4.9 本节小结

本节围绕 OpenClaw 工作区隔离与可信数据空间数据沙箱的融合，提出了一个面向手机智能体记忆治理的“双层边界”架构：OpenClaw 侧负责应用执行边界，可信数据空间侧负责数据使用边界；高敏记忆不再默认跨域复制，而是在受控数据沙箱中按使用控制和数字合约执行分析、聚合与最小必要输出。该架构一方面吸收了 OpenClaw 官方关于“混合信任场景应拆分信任边界”“工作区不是授权边界”“会话标识不是授权令牌”的明确安全结论，另一方面也吸收了可信数据空间关于“数据沙箱、使用控制、数字合约、行为校验、执行存证”的正式要求。

从研究价值看，这一融合方案并不是声称 OpenClaw 当前已默认具备完整的可信数据空间能力，而是在其现有工作区、沙箱和安全边界基础上，提出一条可渐进落地的增强路线：低敏场景可采用进程级或容器级隔离，高敏场景进一步叠加 OpenShell、NemoClaw 或可信执行环境等更强边界，并把跨域使用统一纳入数据沙箱与使用控制链路。这样，系统才能真正做到：数据不自由流动，而是在受控边界内生成可共享的最小必要结果。

### 第 4 章小结

本章围绕手机智能体个性化记忆管理中的隐私保护与合规共享问题，构建了一套贯穿“分类—保护—同步—受控共享”的技术方案。首先，第 4.1 节基于向量检索场景提出五维分类模型和四层隐私分级体系，通过综合敏感度评分函数对记忆对象进行可解释分类；其次，第 4.2 节在此基础上提出分层存储、分级加密、基于角色的访问控制、基于属性的访问控制和使用控制相结合的细粒度保护机制，并通过密钥分域、用途控制、场景隔离和遗忘机制提升静态与动态保护能力；再次，第 4.3 节围绕跨设备同步设计了“本地优先同步 + 云辅助同步”的双模式架构，通过差分隐私、安全聚合和异质化预算分配，在个性化收益与隐私损失之间建立可调平衡；最后，第 4.4 节在 OpenClaw 工作区、沙箱和信任边界的官方安全语义基础上，引入可信数据空间的数据沙箱、使用控制和数字合约能力，形成应用执行边界与数据使用边界相结合的双层边界架构。

从整体上看，第 4 章的技术路线并不是把“记忆能力”和“隐私能力”割裂处理，而是把记忆对象的产生、存储、索引、召回、同步和共享全过程纳入统一治理框架。其核心思想可以概括为：对低敏对象，优先保证可用性；对中敏对象，强化分层保护与用途约束；对高敏对象，默认不自由流动，仅在受控边界内生成最小必要结果。由此，本章为后续的原型验证、实验设计和阶段性成果评估提供了完整的方案基础。

## 5.1 总体架构设计原则

总体技术架构遵循五项基本原则：
**本地优先（local-first）**、**最小权限（least privilege）**、**策略默认生效（policy-by-default）**、**隐私保护内生设计（privacy-by-design）**、**默认可审计（auditable-by-default）**。其中，本地优先强调高敏记忆对象默认保留在用户可控边界内；最小权限强调记忆对象仅在满足角色、属性、用途和场景约束时才可被访问；策略默认生效强调记忆写入、索引、召回、同步和共享均以策略判定为前置条件；隐私保护内生设计强调分级分类、分层存储、差分隐私和数据沙箱等机制从架构层面嵌入系统；默认可审计强调访问、同步、共享、撤销和删除全过程均保留审计证据与履约记录。

本章面向手机智能体个性化记忆治理的全流程落地，目标不是单点提升某一项能力，而是在个性化可用性、隐私保护强度、跨设备连续性与合规共享能力之间建立统一架构。结合 OpenClaw 对工作区、沙箱和信任边界的官方定义，以及可信数据空间技术架构中“数字合约 + 使用控制 + 数据沙箱”的核心思路，本文提出六项总体设计原则：**本地优先、分层治理、最小必要、策略前置、受控流通、全过程可审计**。其中，“本地优先”强调高敏记忆对象默认留在用户可控边界内；“分层治理”强调内容层、检索层和治理层分离；“策略前置”强调在检索、同步和共享之前完成授权与用途判定；“受控流通”强调共享的应是受限结果而非原始记忆；“全过程可审计”强调从写入、检索、同步到撤销都需留下证据链。

从系统安全边界看，OpenClaw 官方明确指出：工作区是智能体的 home，应视为 memory 并保持私密，但它只是默认当前工作目录，并不是硬沙箱；混合信任场景下应拆分信任边界，而不能把单一网关视为天然多租户安全边界。这意味着，总体技术架构必须同时覆盖逻辑边界、运行边界和信任边界三层约束，而不能只依赖“不同工作区目录”或“不同会话标识”来获得安全性。

基于此，本文将总体架构目标抽象为统一约束函数：

$$P_{\text{系统有效}}= P_{\text{分类}} \land P_{\text{存储}} \land P_{\text{访问}} \land P_{\text{同步}} \land P_{\text{沙箱}} \land P_{\text{合约}} \land P_{\text{审计}} \tag{5-1}$$

其中，$P_{\text{分类}}$表示记忆对象已完成分级分类，$P_{\text{存储}}$表示分层存储与分级加密已生效，$P_{\text{访问}}$表示访问控制和使用控制判定通过，$P_{\text{同步}}$ 表示跨设备同步满足差分隐私预算约束，$P_{\text{沙箱}}$表示运行边界满足受控执行要求，$P_{\text{合约}}$表示跨域共享符合数字合约与用途约束，$P_{\text{审计}}$表示全流程可追溯。该式的核心含义是：个性化记忆的可用性必须建立在多重治理条件同时成立的基础上，而非单点技术措施。这个整体思路与可信数据空间“环境安全、执行可信、行为可证”的技术要求是一致的。

## 5.2 系统分层架构

### 5.2.1 总体分层

结合前文第 4 章的研究结果，本文建议系统采用六层架构：

1. 记忆采集与预处理层
2. 记忆治理与分级分类层
3. 存储与检索层
4. 策略执行与权限控制层
5. 跨设备同步与共享层
6. 审计与合规治理层

这样的分层并不是简单的功能拆分，而是围绕风险传播路径进行的边界设计。OpenClaw 的记忆文件、工作区和索引机制为前四层提供了直接工程接口，可信数据空间中的连接器、使用控制、数据沙箱和数字合约则为后两层提供了跨域共享与受控流通的基础能力。

### 5.2.2 各层职责

**第一层：记忆采集与预处理层。**
该层负责从用户输入、系统观察、工具调用结果、外部回填数据中抽取候选记忆对象，并执行基础清洗、去重、敏感实体识别和上下文绑定。其输出对象不直接进入长期记忆，而是先进入分级分类流程。由于 OpenClaw 的长期记忆和日常记忆都显式写入工作区文件，因此这一步适合嵌入记忆写盘前和索引更新前的管道中。

**第二层：记忆治理与分级分类层。**
该层根据五维分类字段，即内容敏感度、来源可信度、使用目的、场景域、生命周期，对记忆对象进行综合评分和 L0—L3 隐私等级划分，并生成策略接口字段。该层决定“能否入索引、能否普通召回、能否同步、能否共享、是否必须沙箱执行”，是全系统的策略入口。其设计依据既来自前文提出的五维分类模型，也符合 IDS 关于“数据使用控制应在数据处理前给出明确约束”的要求。

**第三层：存储与检索层。**
该层将记忆对象拆分为内容层、检索层和治理层：原始片段与摘要片段存放于内容层，向量表示与检索索引存放于检索层，隐私等级、用途约束、场景域和策略版本等存放于治理层。OpenClaw 当前采用 Markdown 记忆文件和 SQLite 索引的设计，使这种分层在工程上具备可操作性。

**第四层：策略执行与权限控制层。**
该层融合基于角色的访问控制、基于属性的访问控制和使用控制，决定谁在什么设备上、什么场景域中、出于什么目的、可以对什么记忆对象执行什么动作。NIST SP 800-162 给出了基于属性的访问控制定义，NIST SP 800-207 则要求在零信任架构中把主体、设备和环境都纳入动态判定。

**第五层：跨设备同步与共享层。**
该层负责把符合前置条件的中低敏记忆对象转化为受控同步输入，并通过本地优先同步或云辅助同步完成跨设备连续性支持。云辅助路径下应采用联邦式更新、差分隐私和安全聚合，避免直接传输原始高敏记忆。Gboard 的公开工业实践已证明，这条路线在移动端语言模型场景中是可行的。

**第六层：审计与合规治理层。**
该层负责统一记录访问、同步、共享、撤销和删除行为，并与可信数据空间中的数字合约、履约存证和全程溯源要求对接。国家数据局技术文件已明确接入连接器和平台都应支持策略解析、行为校验、执行记录和审计追溯。

### 5.2.3 分层架构的统一表达

为统一描述六层架构，本文将系统输出写为：

$$Y = \mathcal{F}(M,\ A,\ \Pi,\ E,\ C)$$

其中，$M$ 表示记忆对象集合，$A$ 表示主体与设备属性集合，$\Pi$表示策略集合，$E$表示运行环境与场景域状态，$C$表示数字合约与用途约束集合，$Y$表示系统最终输出。该式表示：系统输出不再是对原始记忆的直接暴露，而是在记忆对象、策略、环境和合约共同约束下生成的受控结果。

### 5.2.4 记忆数据的三层共享架构

在第 5.2 节前文中，本文已经提出由记忆采集与预处理层、记忆治理与分级分类层、存储与检索层、策略执行与权限控制层、跨设备同步与共享层、审计与合规治理层构成的六层系统架构。为了进一步说明“记忆对象在系统中如何流通”，有必要在六层架构基础上补充一个面向共享路径的抽象，即三层共享架构。该架构并不是对六层系统架构的替代，而是从“共享深度”和“边界范围”的角度对数据流通形态进行归纳，分别包括：用户自主共享、数据空间共享和联邦共享。这一补充的意义在于，它能够把前文分散于第 4.2 节、第 4.3 节和第 4.4 节中的共享机制统一到同一框架中。

**第一层是用户自主共享。** 该层主要发生在单一用户控制边界之内，例如同一用户在手机、平板和电脑等多设备上的受控同步，或用户明确授权某个本地智能体访问其另一工作区中的低敏记忆对象。此类共享的核心特征是：数据控制权仍然由用户直接掌握，共享决策基于显式授权、可撤销授权和用途约束，而不涉及外部组织或数据空间中的多方协作。前文第 4.3 节提出的“本地优先同步”以及第 4.2 节提出的“场景域隔离 + 细粒度访问控制”，都可以视为这一层的技术实现。换言之，用户自主共享是三层共享架构中最基础的一层，其关注点不在于复杂跨组织流通，而在于保障用户在本地连续性与最小暴露之间取得平衡。

**第二层是数据空间共享。** 该层适用于跨组织、跨平台或跨主体的受控协作场景。在这一层中，记忆对象原则上不以“原始整条记忆”形式直接流出，而是按照第 4.4 节提出的双层边界思路，被封装为带治理元数据的共享对象、最小必要片段、统计结果或受限数据产品。其共享过程需通过数据空间连接器、数字合约、使用控制和数据沙箱共同约束。IDS 参考架构和 usage control 公开材料都强调，数据主权要求数据提供方在共享后仍然保留对数据使用方式的控制；TANGO 项目及其白皮书则进一步展示了如何通过数据空间连接器、区块链、自主主权身份和策略执行机制增强这种受控共享的可信度与韧性。对本课题而言，这一层是“记忆数据进入可信数据空间后如何受控流通”的核心路径。

**第三层是联邦共享。** 这一层与前两层最大的区别在于，它不以共享原始记忆对象为目标，而是以共享统计量、模型更新或聚合后的受限表示为目标。Gboard 的工业实践表明，移动端模型可以通过联邦学习在本地完成训练或更新，仅上传受控的模型增量，并配合差分隐私和安全聚合获得形式化保护；这说明在手机智能体场景中，“原始高敏记忆不出端，只让模型或统计结果跨端流动”是一条现实可行路线。对应到本文框架中，这一层主要服务于跨设备记忆连续性、偏好参数更新和轻量模型协同，而不是服务于原始文本级的跨主体共享。它更强调“能力共享而非数据共享”。

从总体逻辑上看，三层共享架构可以理解为一个由“强控制的本地共享”逐步过渡到“受控结果共享”和“联邦更新共享”的层次体系。其约束强度和数据暴露程度具有明显差异：用户自主共享暴露面最小，数据空间共享强调可控使用与可审计流通，联邦共享则通过不直接传递原始数据来进一步降低暴露面。为了形式化这一结构，本文定义共享层级选择函数：

$$S^\ast = \arg\min_{S \in \{\text{用户自主共享},\ \text{数据空间共享},\ \text{联邦共享}\}} \Bigl\{ \operatorname{共享风险}(m,d,p) \le \operatorname{能力上界}(S) \Bigr\}$$

其中，$m$ 表示待共享记忆对象集合，$d$ 表示场景域，$p$ 表示使用目的。该式的含义是：对某一类记忆对象和某一类用途，应优先选择满足业务需求的最低暴露共享层级。也就是说，若用户自主共享已经足够，就不应上升到数据空间共享；若联邦共享即可支持模型协同，就不应暴露原始记忆内容。这个逻辑与前文提出的“最小必要输出”和“受控边界内共享”完全一致。

进一步地，三层共享架构也为第 5 章的六层系统架构提供了更清晰的数据流解释：用户自主共享主要依托记忆治理层、权限控制层与同步层完成；数据空间共享主要依托受控共享模块、数字合约管理模块和审计中心完成；联邦共享则主要依托同步控制模块、差分隐私处理模块和安全聚合模块完成。由此，第 5 章的总体技术架构不仅解释了“系统有哪些层”，也进一步解释了“记忆数据在不同风险级别下如何选择不同共享路径”。这将有助于后续若扩展“记忆数据资产化与合规流通”章节时，直接在三层共享架构之上进一步讨论计量、定价和收益分配，而无需重新组织技术主线。

### 5.3.1 记忆采集模块

记忆采集模块负责把外部输入转化为标准化记忆对象，包括用户显式输入、工具执行结果、外部服务回填、系统总结与上下文观察。模块输出应包含至少五个字段：对象内容、对象类型、来源、时间戳、上下文边界。之所以要保留来源和上下文，是因为后续分级分类和用途控制并不只取决于文本内容本身，还取决于“这条记忆是用户明确说的，还是系统推断出来的”。这与前文的五维分类模型保持一致。

### 5.3.2 分级分类模块

分级分类模块负责实现综合敏感度评分，并输出 L0—L3 隐私等级及五维分类字段。其内部可进一步拆为三个子模块：

- 一是显式敏感识别子模块，识别身份、联系方式、财务、健康、位置等显式敏感实体；
- 二是语义风险识别子模块，识别隐含偏好、行为模式和跨场景可重识别线索；
- 三是策略映射子模块，把分类结果转化为可执行的策略接口字段。

### 5.3.3 存储与检索模块

该模块对应前文分层存储架构，负责把记忆对象写入内容层、检索层和治理层，并在写入时附带分级密钥、场景域和用途限制。对 OpenClaw 而言，这意味着长期记忆文件、近端记忆文件与索引数据库不应再被视为“普通文件集合”，而应被纳入统一治理对象。由于工作区默认不是硬沙箱，该模块必须与沙箱执行配置和工作区访问权限联动。

### 5.3.4 策略决策模块

策略决策模块是整个系统的实时治理核心，负责根据角色、属性、场景域、用途、数字合约和运行环境做出允许、拒绝或进一步验证三类决策。其输出不仅影响记忆读取，还影响是否能同步、是否能跨域共享、是否必须在受控沙箱中执行。该模块的设计依据是 NIST 的基于属性的访问控制与零信任原则，以及 IDS 的使用控制思想。

### 5.3.5 同步控制模块

同步控制模块负责筛选可同步对象、分配差分隐私预算、执行本地裁剪和加噪、管理安全聚合协议，并在下发阶段再次校验场景域和用途限制。其核心不是“复制记忆”，而是“复制在策略约束下允许被复制的受限表示”。这与 Gboard 的联邦式更新实践是一致的。

### 5.3.6 受控共享模块

受控共享模块负责将高敏跨域处理统一收口到数据沙箱和数字合约框架下。模块输入不是“任意外部调用”，而是必须满足跨域有效条件的正式请求；模块输出不是原始记忆，而是最小必要片段、统计结果或受限数据产品。这一模块直接对应可信数据空间技术架构中“数字合约 + 使用控制 + 数据沙箱 + 行为校验”的组合要求。

## 5.4 数据流与控制流设计

[图片]

### 5.4.1 数据流设计

本系统的数据流可分为四条主链路：写入链路、检索链路、同步链路、共享链路。

- **写入链路**：候选记忆对象进入采集模块后，经分级分类模块判定隐私等级与治理属性，再进入内容层、检索层和治理层分层写入。该链路的核心约束是“未分类不入库，未赋权不入索引”。
- **检索链路**：请求主体首先经过策略决策模块判定是否允许读取目标记忆对象；若允许，再根据隐私等级决定是否走普通召回、受限召回或沙箱内处理。对高敏对象，应避免把原始记忆直接加载到普通上下文中。
- **同步链路**：同步控制模块先根据前文定义的同步可行性函数筛选对象，再执行裁剪、加噪、安全聚合和受控下发。整个过程中，策略接口字段“是否允许跨设备同步”是关键门槛。
- **共享链路**：任何跨场景、跨主体共享都需先进入受控共享模块，由数字合约、使用控制和数据沙箱共同判定是否允许处理，并只输出最小必要结果。

可将全局数据流抽象为：

$$M_{\text{输出}} = \Psi\bigl(M_{\text{输入}},\ \Pi_{\text{分类}},\ \Pi_{\text{权限}},\ \Pi_{\text{同步}},\ \Pi_{\text{共享}}\bigr) \tag{5-3}$$

其中，$M_{\text{输入}}$表示初始记忆对象集合，$\Pi_{\text{分类}}$、$\Pi_{\text{权限}}$、$\Pi_{\text{同步}}$、$\Pi_{\text{共享}}$ 分别表示分类、权限、同步和共享策略集合，$\Psi$表示系统对记忆对象的全流程变换。该式强调：输出对象是经过多层策略裁剪后的结果，而不是原始输入的简单复制。

### 5.4.2 控制流设计

与数据流并行的，是贯穿全系统的控制流。控制流的核心是：任何会改变记忆可见范围、传播范围或用途边界的动作，都必须先经过策略判定，再执行操作，最后留下审计记录。因此，控制流包括三个连续阶段：

1. **策略解析阶段**：解析隐私等级、角色、属性、场景域、数字合约和生命周期状态。
2. **执行判定阶段**：输出允许、拒绝或进一步验证。
3. **证据留存阶段**：记录本次决策依据、执行动作和结果状态。

这一链路与可信数据空间使用控制技术文件中的“策略解析—行为校验—操作执行—行为记录存证”结构是一致的。

可将控制流抽象为：

$$R = \Omega(r,\ A_s,\ A_o,\ A_e,\ \Pi,\ C) \tag{5-4}$$

其中，$r$表示请求，$A_s$、$A_o$、$A_e$分别表示主体属性、客体属性和环境属性，$\Pi$ 表示策略集合，$C$ 表示数字合约与用途约束，$R$ 表示决策结果。该结果可取“允许”“拒绝”“进一步验证”三类状态。这个式子把第 4.2 节的访问控制模型和第 4.4 节的跨域行为校验统一到了同一框架中。

[图片]

> **策略中心、权限中心与审计中心联动图。** 该图从治理视角展示了系统中三类核心中心的分工与协同关系：策略中心负责汇聚隐私等级、五维分类字段、数字合约和组织规则，形成可执行策略并下发至各模块；权限中心负责对记忆采集、分类、存储检索、同步、共享、撤销等请求进行实时授权与用途判定；审计中心负责汇聚运行日志、生成证据链并反哺策略优化，从而形成“规则生成—执行判定—证据留存—策略更新”的闭环。

### 5.5.1 策略中心

策略中心是全系统的治理规则汇聚点，负责维护隐私等级规则、场景域规则、用途规则、同步规则、共享规则和撤销规则。其输入来自分级分类结果、数字合约和组织策略，其输出则是可被权限中心和执行模块直接调用的策略接口字段。策略中心必须支持版本化、可解释和可回滚，否则一旦策略发生漂移，后续审计将失去依据。可信数据空间数字合约技术文件也明确指出，数字合约管理应覆盖创建、协商签署、履行、备案、争议解决和解除等全生命周期。

形式化地，策略中心可视为生成策略集的函数：

$$\Pi^\ast = \Gamma(L,\ D,\ U,\ C,\ T) \tag{5-5}$$

其中，$L$ 表示隐私等级，$D$表示场景域，$U$ 表示使用目的，$C$ 表示数字合约约束，$T$ 表示生命周期状态，$\Pi^\ast$ 表示当前生效策略集。

### 5.5.2 权限中心

权限中心负责对实时请求执行授权判定，是策略中心的执行代理。它根据角色边界、属性匹配、环境可信度、工作区访问权限、沙箱状态和密钥可用性，决定请求是否可执行。NIST 的基于属性的访问控制与零信任文档都支持这种“角色边界 + 属性动态校验 + 环境状态”的组合模式。

权限中心输出的不是简单读写许可，而应至少覆盖以下五个统一字段：
是否允许索引、是否允许普通召回、是否允许跨设备同步、是否允许外部共享、是否强制沙箱执行。这五个字段与前文各节保持一致。

### 5.5.3 审计中心

审计中心是整个架构中“可解释、可追溯、可问责”的支撑模块。它应统一收集访问日志、策略判定日志、同步日志、共享与撤销日志，并为后续合规核验提供证据链。国家数据局使用控制技术要求与数字合约技术要求都把行为记录、查询审计、履约存证作为关键能力。

审计中心可抽象为审计证据生成函数：

$$E^\ast = \Lambda(r,\ R,\ \Pi^\ast,\ t) \tag{5-6}$$

其中，$r$ 表示请求，$R$ 表示决策结果，$\Pi^\ast$ 表示生效策略，$t$ 表示时间戳，$E^\ast$表示本次操作生成的审计证据。该式强调：审计不是对最终结果的被动记录，而是决策上下文的完整留存。

## 5.6 本章小结

本章在前文研究基础上，进一步提出了手机智能体个性化记忆治理的总体技术架构与方案设计。整体上，系统以“本地优先、分层治理、最小必要、策略前置、受控流通、全过程可审计”为设计原则，构建了由记忆采集与预处理层、记忆治理与分级分类层、存储与检索层、策略执行与权限控制层、跨设备同步与共享层、审计与合规治理层组成的六层架构。该架构一方面以 OpenClaw 的记忆文件、工作区、沙箱和信任边界为现实工程基础，另一方面以可信数据空间的数字合约、使用控制、数据沙箱和履约存证为跨域共享支撑，并以 NIST 的基于属性的访问控制和零信任原则作为动态授权依据。

从技术逻辑上看，本章实现了对前述各节方案的统一收束：记忆对象先经分类，再进入分层存储和权限控制；满足条件的对象才能进入差分隐私保护下的跨设备同步；高敏跨域处理最终统一收口到受控数据沙箱与数字合约链路中执行。这意味着，系统不再以“记忆自由流动”为默认前提，而是以“记忆在受控边界内按策略生成最小必要结果”为核心目标。这样，第 5 章既为后续原型实现提供了清晰架构，也为第 6 章的可行性验证与实验设计提供了结构化落点。

### 6.1.1 原型验证目标与实现边界

本章原型的目标不是直接构建一个面向生产环境的完整系统，而是验证第 4 章和第 5 章提出的关键机制在工程上是否可闭环：即记忆对象能否被分级分类、能否按等级进入分层存储与受控检索、能否在不直接同步原始高敏记忆的前提下支持跨设备连续性、以及能否在跨场景条件下通过双层边界实现最小必要共享。由于 OpenClaw 官方已明确说明其默认信任模型是“个人助理/单一受信操作者边界”，不宜把一个共享网关直接视为敌对多租户安全边界，因此本章原型把“混合信任场景的强隔离”定义为增强验证项而非默认条件：基线原型以单用户、多场景分区为主，增强原型再验证拆分网关、拆分操作系统用户和沙箱强化。

从工程实现角度，原型采用“OpenClaw 记忆工作流 + 分级分类中间层 + 策略执行层 + 同步与共享中间层”的最小可运行架构。其核心思想是：不重写 OpenClaw 的记忆机制，而是在记忆写入、记忆检索、跨设备同步和跨域共享这四个关键节点插入受控中间层。这样做的理由有两点：第一，OpenClaw 官方已提供记忆文件、内建索引和沙箱配置这些现实接口，适合作为验证宿主；第二，这种方式能更准确反映“现有系统基础上做安全增强”的可行性，而不是构造一个脱离实际产品边界的理想系统。

### 6.1.2 原型总体组成

原型系统由六个验证模块组成：

1. **记忆治理代理层**。该层位于 OpenClaw 记忆写入前和检索前，负责执行第 4.1 节定义的五维分类、L0—L3 隐私等级判定和策略接口字段生成。其输入是候选记忆对象，输出是带有治理元数据的受控记忆对象。
2. **分层存储层**。该层把原始片段、摘要片段、向量表示、检索索引和治理元数据分开管理，并执行第 4.2 节的分级加密与删除同步控制。对原型而言，重点不在于引入最复杂的加密后端，而在于验证“内容层/检索层/治理层分离 + 权限前置 + 删除联动”是否能工作。
3. **权限与使用控制层**。该层基于基于角色的访问控制、基于属性的访问控制和使用控制的组合模型，对检索、同步、共享和删除请求给出“允许、拒绝、进一步验证”三类结果。NIST 对基于属性的访问控制和零信任的定义，适合作为该模块的设计依据。
4. **同步控制层**。该层负责筛选可同步记忆对象，对其中的中低敏对象执行裁剪、加噪、安全聚合模拟和受控下发。原型中不要求复现完整工业级联邦训练，但应至少验证“低敏摘要/偏好同步 + 中敏统计同步 + 高敏原文不默认同步”这一机制是否可行。Gboard 已证明联邦学习与差分隐私可在移动端语言模型中实现形式化保证与实际部署，因此本原型的同步设计是有现实参考的。
5. **跨域受控共享层**。该层负责把高敏跨场景处理统一收口到数据沙箱模拟环境中，只输出统计结果、最小必要片段或受限数据产品，而不直接输出原始高敏记忆。国家数据局的可信数据空间技术架构和使用控制技术要求都将数据沙箱、数字合约、行为校验和履约存证列为关键能力。
6. **审计与撤销层**。该层记录访问日志、策略判定日志、同步日志、共享与撤销日志，并验证删除后内容层、检索层和治理层的联动失效是否生效。该模块直接对应第 4.2 节与第 4.4 节的“全过程可审计”目标。

### 6.1.3 原型实现级别划分

为了避免一次性验证过多机制导致原型不可控，本文建议把原型分成三个级别。

- **一级原型（基线可运行）**：验证记忆分级分类、分层存储、策略前置检索和基础审计。
- **二级原型（同步增强）**：在一级原型基础上增加跨设备同步与差分隐私预算管理。
- **三级原型（跨域增强）**：在二级原型基础上增加数据沙箱、数字合约和最小必要输出验证。

这种渐进式设计与 OpenClaw 官方推荐的“先建立清晰运行边界，再逐步增强能力”的工程思路是一致的，也能把风险控制在可测试范围内。

> **图 6-2 验证目标与实验问题映射图**。该图展示了本章四类验证目标与六个核心实验问题之间的对应关系。

[图片]

### 6.2.1 总体验证目标

本章验证聚焦四类目标：

1. **治理有效性**：验证五维分类字段和 L0—L3 隐私等级是否足以支撑后续存储、访问、同步和共享控制。
2. **个性化保持性**：验证在引入加密、权限控制、差分隐私和沙箱边界后，系统是否仍能保持足够的任务连续性和个性化收益。
3. **安全鲁棒性**：验证系统在成员推断、记忆投毒、越权召回和跨场景串扰等风险下的防护效果。
4. **合规可解释性**：验证系统是否能对“为什么允许、为什么拒绝、为什么只输出受限结果”给出可审计解释。

这些目标分别对应第 4 章各节的技术主线，也与 NIST 对差分隐私评估、OpenClaw 对沙箱与边界配置、以及 RAG/记忆库攻击研究的关注点一致。

### 6.2.2 核心实验问题

围绕上述目标，本文提出六个核心实验问题：

- **问题一**：记忆分级分类能否稳定区分低敏、中敏和高敏对象，并与实际控制策略保持一致？
- **问题二**：分层存储和细粒度访问控制是否能显著降低越权召回率和跨场景串扰率？
- **问题三**：差分隐私与安全聚合引入后，跨设备同步在多大程度上牺牲了个性化连续性与召回质量？
- **问题四**：数据沙箱和最小必要输出机制能否在不直接输出原始高敏记忆的前提下满足跨域任务需求？
- **问题五**：针对成员推断和记忆投毒等攻击，所提机制是否能够降低攻击成功率或限制攻击后果？
- **问题六**：审计、撤销和删除链路是否完整，能否支撑追溯和可解释性要求？

这里，第五类问题尤其重要，因为已有研究表明 RAG 外部数据库可能受到成员推断攻击，而 AgentPoison 证明了少量投毒即可让依赖记忆或知识库的智能体在保持正常性能的同时出现高攻击成功率。

### 6.3.1 数据集设计

为了兼顾可复现性与合规性，本文建议采用“三类数据集”设计。

- **第一类：合成记忆数据集。** 基于人工构造或脚本生成的记忆对象，覆盖一般偏好、任务习惯、地点关系、日程约束、财务线索、身份线索、工作上下文等多种类型，并为每条对象预先标注五维分类字段和隐私等级。其优点是边界清晰、便于做精确评价，适合作为分级分类和策略映射的基础测试集。
- **第二类：受控真实语料改写数据集。** 从公开可用文本、公开任务数据或研究用工作流语料中提取内容，再经过脱敏、重写和结构化处理，构造成“近似真实”的记忆对象集合。该数据集更接近实际语言分布，适合测试分类模型、检索行为和摘要同步策略的泛化能力。
- **第三类：对抗测试数据集。** 用于成员推断、越权召回、记忆投毒、跨场景串扰和删除失效验证。成员推断测试可借鉴 RAG 相关研究的思想，以“目标样本是否存在于记忆库中”为判别任务；投毒测试则可参考 AgentPoison 的思路，构造极少量恶意记忆样本并观察其被召回和影响行为的概率。

### 6.3.2 任务集设计

任务集应覆盖“记忆写入—记忆召回—跨端同步—跨域共享”四条主链路。建议至少包含以下五类任务：

1. 个性化问答任务：验证系统能否根据长期偏好和近期上下文提供连续回答。
2. 跨会话任务承接任务：验证系统在不同会话间能否保持待办状态、决策上下文和偏好约束。
3. 跨设备恢复任务：验证不同设备能否在不同步原始高敏记忆的前提下恢复必要上下文。
4. 跨域最小必要共享任务：验证工作域或第三方域能否在只获得受限结果的情况下完成业务需求。
5. 删除/撤销后恢复任务：验证对象被删除或撤销共享后，系统是否仍会错误召回、错误同步或错误共享。

这些任务与 OpenClaw 记忆和检索机制的真实工作方式相符，因为其长期记忆与日常记忆本身就是为跨会话连续性和持续助手能力设计的。

### 6.3.3 场景集设计

场景集应直接对应前文的场景域设计，至少包含三类：

- **个人域场景**：重点测试私人偏好、日程和高敏个人记忆的本地保留与受控使用。
- **工作域场景**：重点测试项目上下文、会议事项与组织流程记忆在角色边界和用途限制下的访问与同步。
- **第三方域/共享域场景**：重点测试外部服务或可信数据空间参与方如何通过数据沙箱只获得最小必要结果。

此外，建议增加一个混合信任场景作为增强验证项：在两个独立网关或两个独立操作系统用户下，分别部署个人域和工作域，验证拆分信任边界后串扰率、越权率和审计清晰度是否得到改善。这一设计直接呼应 OpenClaw 官方关于 mixed-trust 部署应拆分 trust boundaries 的建议。

### 6.4.1 个性化与功能指标

为衡量系统在引入保护机制后是否仍然“有用”，本文建议定义综合可用性指标：

$$U_{\text{综合}} = \alpha U_{\text{连续性}} + \beta U_{\text{个性化}} + \gamma U_{\text{任务完成}} $$

其中，$U_{\text{连续性}}$ 衡量跨会话或跨设备的上下文承接能力，$U_{\text{个性化}}$ 衡量回答或决策对稳定偏好的遵循程度，$U_{\text{任务完成}}$ 衡量任务是否成功完成，$\alpha,\beta,\gamma$ 为权重。该指标用于评价安全机制引入前后的效用变化。

与之配套，可定义效用损失：

$$ \Delta U = U_{\text{基线}} - U_{\text{保护后}} $$

若 $\Delta U$ 过大，则说明当前保护策略虽增强了隐私，但已显著削弱了系统实用性。

### 6.4.2 隐私与安全指标

本章建议至少使用以下四类安全指标。

**越权召回率：**

$$R_{\text{越权}} = \frac{N_{\text{越权召回}}}{N_{\text{总召回请求}}} \tag{6-3}$$

用于衡量本不应被当前主体或场景域读取的记忆被错误召回的比例。

**敏感泄露率：**

$$L_{\text{泄露}} = \frac{N_{\text{泄露敏感对象}}}{N_{\text{敏感对象总数}}} \tag{6-4}$$

用于衡量高敏对象在检索、同步、共享或日志链路中被非预期暴露的比例。

**攻击成功率：**

$$ASR = \frac{N_{\text{攻击成功样本}}}{N_{\text{攻击总样本}}} \tag{6-5}$$

用于衡量成员推断、投毒触发、越权调用等攻击的成功概率。该指标可直接用于对比引入防护前后的攻击压制效果。AgentPoison 的研究采用平均攻击成功率作为核心指标，因此本章沿用这一思路是合理的。

**跨场景串扰率：**

$$R_{\text{串扰}} = \frac{N_{\text{跨域误访问}}}{N_{\text{跨域访问总请求}}} \tag{6-6}$$

用于衡量个人域、工作域和第三方域之间因边界不清而发生的记忆误访问比例。

### 6.4.3 差分隐私与同步指标

针对跨设备同步，建议增加三类指标。

**累计隐私损失**：使用 Rényi 差分隐私会计或等价方法累加各轮损失，验证最终是否满足预设的 $(\varepsilon,\delta)$ 目标。NIST SP 800-226 明确指出，评估差分隐私保证时必须重视组合、预算与威胁模型的一致性。

**同步精度损失：**

$$\Delta S = S_{\text{无噪声同步}} - S_{\text{差分隐私同步}} \tag{6-7}$$

其中，$S$ 表示跨设备同步后对目标任务的恢复精度或偏好一致性。

**通信与能耗成本：**

$$C_{\text{总代价}} = \lambda_1 C_{\text{通信}} + \lambda_2 C_{\text{时延}} + \lambda_3 C_{\text{能耗}} \tag{6-8}$$

用于衡量差分隐私和安全聚合引入后的实际设备代价，避免方案只在理论上可行而在移动端代价过高。

### 6.4.4 可审计与可撤销指标

为了验证“合规共享”不是停留在访问控制层，建议引入：

**审计完整度：**

$$A_{\text{完整}} = \frac{N_{\text{已记录关键事件}}}{N_{\text{关键事件总数}}} \tag{6-9}$$

**撤销生效率：**

$$R_{\text{撤销}} = \frac{N_{\text{撤销后失效对象}}}{N_{\text{撤销对象总数}}} \tag{6-10}$$

**删除一致性：**

$$D_{\text{一致}} = \frac{N_{\text{内容层、检索层、治理层同步清理对象}}}{N_{\text{删除请求对象总数}}} \tag{6-11}$$

这些指标直接对应第 4.2 节和第 4.4 节提出的“全过程可审计、可撤销、可删除”目标。

> **图 6-4 可行性验证路线图**。该图展示了本章四条验证路线及其对应的核心观测指标。

[图片]

### 6.5.1 验证路线一：分类—控制闭环验证

第一条验证路线聚焦前两节提出的治理闭环：
先对候选记忆对象执行五维分类和 L0—L3 分级，再根据策略接口字段控制是否允许索引、是否允许普通召回、是否允许同步、是否允许共享。实验上，使用合成记忆数据集和受控真实语料改写数据集，分别评估分级准确性、越权召回率、串扰率和效用损失。该路线的目标是回答“分类是否真的能支撑控制，而不是停留在标签层”。

### 6.5.2 验证路线二：同步—隐私权衡验证

第二条路线聚焦第 4.3 节提出的同步保护机制。设置至少三组对照：
- 一组为无同步保护基线，直接传输中低敏摘要或受限表示；
- 一组为差分隐私同步组，加入裁剪与加噪；
- 一组为差分隐私 + 安全聚合组，进一步隐藏单端贡献。

比较三组在个性化连续性、同步精度损失、累计隐私预算、通信成本和攻击成功率上的差异。Gboard 的工业实践说明，这类对照是合理的，因为其公开经验就是围绕“形式化隐私保证与效用折中”展开。

### 6.5.3 验证路线三：跨域受控共享验证

第三条路线聚焦第 4.4 节的双层边界方案。实验设置三组跨域共享模式：
- 一组是直接原文共享基线；
- 一组是策略约束下的脱敏或摘要共享；
- 一组是数据沙箱中的最小必要输出。

比较三组在任务完成率、敏感泄露率、审计完整度和可解释性上的差异。该路线的关键不是“哪个模式最方便”，而是验证最小必要输出是否能在不共享原始高敏记忆的条件下保留足够业务价值。国家数据局关于数据沙箱和数字合约的技术文件支持这种设计方向。

### 6.5.4 验证路线四：对抗与红队验证

第四条路线聚焦攻击面验证。建议至少进行三类红队测试：
- 成员推断测试：针对记忆库或检索数据库，测试外部主体是否能推断某一目标样本是否存在。
- 投毒测试：参考 AgentPoison 思路，向长期记忆或检索库注入少量恶意样本，观察其被召回和影响任务结果的概率。
- 混合信任串扰测试：在单一网关与拆分边界两种部署方式下，对比跨场景串扰率和越权率。

这一路线的价值在于验证“系统在正常任务上表现好”不等于“系统面对攻击仍然可控”。RAG 成员推断研究和 AgentPoison 都已说明，检索增强和长期记忆是现实攻击面。

## 6.6 风险与限制说明

### 6.6.1 工程限制

第一，原型系统的验证结果不能直接等同于生产环境结论。尤其在 OpenClaw 默认信任模型下，如果不拆分网关、操作系统用户或宿主，混合信任场景的隔离强度本身就受限，因此本章对多场景、多主体的验证必须明确说明其边界。

第二，差分隐私预算、裁剪阈值和同步频率都高度依赖任务类型和设备约束，实验中的最佳配置不一定能直接迁移到所有移动端场景。NIST SP 800-226 也强调，差分隐私承诺必须与具体威胁模型、数据类型和系统实现共同评估。

第三，数据沙箱和可信执行环境的增强验证可证明方向可行，但其部署成本、集成复杂度和性能代价也更高，因此本章的重点仍应放在“最小可运行基线 + 可扩展增强路线”上，而不应把所有高强度隔离能力都视作基线要求。

### 6.6.2 方法限制

第一，合成数据集能提供清晰标注，但和真实复杂语言分布仍存在差距，因此必须辅以受控真实语料改写数据集。

第二，成员推断和投毒测试只能覆盖当前已知攻击路径，不能穷尽未来未知攻击。

第三，本章提出的可用性指标和权重带有项目导向性，不同组织或产品可能需要重新设定权重。

### 6.6.3 结果解释限制

本章所有实验更适合回答“该方案是否具有工程可行性”“在哪些条件下有效”“代价有多大”，而不应被过度解读为“已经证明长期个性化记忆系统在所有场景下安全可靠”。尤其是对跨场景共享、对抗攻击和高敏数据处理，验证结果应与 OpenClaw 官方边界声明和可信数据空间的合规要求一起解释，而不能脱离部署前提单独下结论。

## 6.7 本章小结

本章从原型系统设计、实验问题提出、数据集与任务集设计、指标体系构建到可行性验证路线，形成了面向手机智能体个性化记忆治理的验证闭环。整体上，原型验证并不追求一次性证明所有机制都已生产级成熟，而是强调通过渐进式原型和多维实验回答四个问题：分级分类是否有效、保护机制是否可执行、同步与共享是否可控、系统是否仍保持足够个性化收益。OpenClaw 的记忆和沙箱机制、Gboard 的差分隐私联邦学习实践、NIST 的差分隐私评估指南，以及针对 RAG 与长期记忆的成员推断和投毒研究，共同为本章提供了现实参照。

从报告结构上看，本章完成了从“问题定义—技术方案—总体架构”走向“可验证原型—可测指标—可复现实验”的过渡，也为后续成果总结与实施建议提供了证据基础。换言之，只有把前文提出的分类、加密、访问控制、差分隐私和数据沙箱机制真正放进统一原型和统一实验框架中检验，整套方案才具备从研究设想到工程落地的说服力。
