# 技术分享追问口径

更新日期：2026-04-27  
配套正文：[tech-sharing-report.md](./tech-sharing-report.md)、[PPT 提纲](./tech-sharing-slides.md)

## 1. 这和普通权限控制有什么区别？

普通权限控制主要回答“谁能不能读”。本方案更强调 usage control，回答的是“在什么场景、出于什么用途、以什么输出形态、能不能继续同步或转发”。

记忆场景里，只做读取权限不够。因为一条记忆被召回后，还可能被模型总结、被工具转发、被第三方保存、被跨设备同步。记忆治理必须覆盖召回、输出、同步、撤销和审计。

## 2. 为什么不直接把记忆都加密？

加密主要保护存储态数据，不能解决被授权程序或检索链路把数据拿出来以后怎么使用的问题。

长期记忆的风险往往发生在“合法召回后的误用”：跨域召回、第三方导出、摘要泄露、撤销后 stale recall。加密是必要底座，但不能替代检索前策略、输出形态控制和同步策略。

## 3. 为什么强调 chunk，而不是文件级治理？

真实实验里，13 个 OpenClaw 记忆文件在文件级全部被判为高敏，文件级高敏率是 `1.0`；切成 145 个 chunk 后，chunk 级高敏率是 `0.1586`。

这说明整文件治理粒度太粗，会把大量低风险内容连带拦住。chunk 不是为了工程复杂度而复杂，而是为了把普通可用内容和高敏受控内容分开。

## 4. pre-guard 和 post-filter 到底差在哪里？

Post-filter 是内容已经被召回之后再过滤，raw candidate 已经跨过安全边界。Pre-guard 是在内容进入模型上下文或外部消费者之前做策略判定。

实验里，`post_filter` 的 raw boundary 暴露率仍是 `0.875`，而 `pre_guard` 降到 `0.125`；高敏原文暴露率从 `0.5` 降到 `0.0`。所以二者不是等价方案。

## 5. 摘要是不是天然安全？

不是。摘要如果保留敏感实体、关系线索或唯一行为模式，仍可能泄露。

输出形态实验里，`summary` 的原文暴露率是 `0.0`，但敏感泄露率仍有 `0.5`，最小必要输出合规率只有 `0.5`。所以分享时要强调：可用不可见不是“简单摘要”，而是按风险控制输出形态，包括 redaction、derived result、sandbox job 和 deny。

## 6. 为什么可用不可见比简单拒绝更好？

简单拒绝当然安全，但会让个性化能力下降。实验里 `deny` 的任务成功率是 `0.0`，而 `derived_result` 和 `sandbox_job` 在不暴露原文的情况下，任务成功率是 `1.0`，效用分是 `0.8333`。

这说明高敏记忆可以通过受控计算参与任务，只输出最小必要结果。

## 7. 受控流动为什么不等于 summary sync？

Summary sync 能降低原文风险，但不自动解决生命周期和撤销问题。

本地双设备实验里，`summary_sync` 的重识别风险是 `0.0`，但撤销后 stale recall 仍是 `1`；`policy_sync` 同样保持任务成功率 `1.0`，重识别风险 `0.0`，并把撤销后 stale recall 降到 `0`。

所以受控流动至少要同步摘要、策略元数据和 tombstone，必要时再用 DP update。

## 8. 没有真实双设备，当前同步实验可信吗？

当前实验可信的是“同步语义”，不是硬件或网络隔离。

单机双目录可以验证 payload 形态、策略元数据、撤销传播、stale recall、重识别风险等机制。Docker 或真实双设备适合下一阶段验证文件系统隔离、网络边界、挂载权限和不同运行时之间的数据不可见性。

## 9. 攻击测试只有 10 条，会不会太少？

样本确实偏小，不能宣称覆盖完整攻击面。当前攻击集的价值是做方向性压力测试，覆盖 prompt injection、跨域召回、第三方导出、metadata exfiltration、role confusion、workspace boundary probe 和良性保真查询。

汇报时应说“当前攻击集验证了机制收益”，不要说“已经完整防御所有攻击”。

## 10. 分类准确率现在够吗？

当前分类 gold set 规模偏小，已有结果只能说明原型规则在当前样本上可行，不能说明泛化稳定。更严谨的下一步是扩大 gold set，加入多 agent、多工作区、多主题、多语言和更强对抗样本。

主报告里可以把分类准确率放在支撑材料，不建议作为最强主卖点。最强主卖点应是四个治理机制及其 before/after 证据。

## 11. 这套方案的性能开销会不会太大？

当前原型开销可控，但仍需更大规模验证。

汇报数据包中，真实 chunk baseline retrieval p50 是 `2.224 ms`，guarded retrieval p50 是 `2.409 ms`，policy eval p50 是 `0.011 ms`，sandbox overhead p50 是 `2.518 ms`。这些数字说明策略判定本身很轻，但真实生产环境还要评估索引规模、并发、真实 sandbox runtime 和移动端资源限制。

## 12. 这和 OpenClaw 自带 workspace / sandbox 的关系是什么？

OpenClaw 的 workspace 和 sandbox 是重要基础，但不是完整记忆治理。

Workspace 解决记忆文件的组织边界，sandbox 解决工具执行边界；本方案补的是记忆对象生命周期中的治理语义：分类、用途、域隔离、输出形态、同步策略、撤销和审计。

更准确的说法是：不是替代 OpenClaw，而是在 OpenClaw 外显记忆机制之上加一层治理平面。

## 13. 为什么说这是数据资产，而不是普通上下文？

因为长期记忆具备资产特征：

- 可持久化保存
- 可被索引和复用
- 可跨任务产生价值
- 可同步和共享
- 有隐私风险和合规边界
- 需要生命周期管理和审计证明

一旦记忆有这些特征，它就不能再只按 prompt context 处理，而应该按数据资产治理。

## 14. 这套方案最大的创新点到底是什么？

不是某一个单点算法，而是一条治理链路：

```text
记忆资产化
  -> 检索前使用控制
  -> 高敏输出形态控制
  -> 受控同步和撤销传播
  -> 审计可证
```

它把长期记忆从“检索增强能力”推进到“可治理、可验证、可流通的数据资产”。

## 15. 产品化下一步怎么走？

建议分三步：

1. **治理平面接线**：接入真实 OpenClaw memory search hook，让 policy decision 成为默认召回路径的一部分。
2. **高敏执行隔离**：把规则级 sandbox 替换或补充为真实 runtime sandbox，评估容器、系统沙箱或 TEE 的边界。
3. **同步与授权界面**：把 policy sync、tombstone、用户授权、撤销和审计日志做成可观察能力。

这三步完成后，方案才更接近可演示产品，而不只是实验原型。

## 16. 汇报时哪些话不要说？

避免过度表述：

- 不说“彻底解决隐私问题”。
- 不说“完全防止记忆泄露”。
- 不说“差分隐私同步已生产可用”。
- 不说“沙箱已经是真实 TEE 隔离”。
- 不说“当前分类模型已经泛化可靠”。

稳妥表达：

> 当前原型验证了治理机制的方向有效，生产化还需要真实接线、真实沙箱、真实多设备和更大规模样本。
