从能力到治理02 / 19
OpenClaw 作为验证对象

让记忆可工程化，也让风险可放大

03

机会

  • 记忆外显为工作区文件，便于观察和分类。
  • chunk、SQLite、FTS 和向量检索让治理点可接入。
  • 检索、策略、审计可以形成可复现实验链路。
  • 沙箱机制可作为高敏受控分析的基础。
同时

风险

  • workspace 不是天然硬隔离边界。
  • 检索默认偏向相关性，不天然理解用途和权限。
  • 多 agent、多 workspace 容易出现域边界混淆。
  • 第三方接入会放大原文流动和越权召回风险。
OpenClaw 把长期记忆外显化，因此我们可以把“隐私保护”落到可测的工程对象上。
OpenClaw Memory Workspace03 / 19
核心矛盾

更懂用户 vs 更少暴露用户

04

更懂用户

  • 记住长期偏好和使用习惯。
  • 保持任务、日程和工作上下文连续。
  • 跨设备延续个性化体验。
  • 在授权范围内复用记忆价值。
VS

更少暴露用户

  • 个人域、工作域、第三方域隔离。
  • 用途限定和最小必要输出。
  • 原文不默认进入普通上下文。
  • 同步、撤销和审计可证明。
目标不是取一端，而是让记忆参与任务时始终带着边界。

创新点三：可用不可见

高敏记忆不是“给或不给”的二选一

11

raw

可用性最高，但原文暴露不可接受。

summary

降低原文暴露，但摘要不天然安全。

redacted

去除实体或敏感字段，保留任务线索。

derived

只输出派生结果，不返回原始记忆。

sandbox

在受控环境中计算，只放出最小结果。

deny

风险不满足条件时直接拒绝。

错误的二选一

全给：可用但失控。
全拒绝：安全但不可用。

更好的中间层

让记忆参与计算，
但只输出最小必要结果。

边界与下一步

当前是治理层原型，不是生产级完整系统

15

当前边界

  • 真实文件 13 个，真实 chunk 145 个，样本仍偏小。
  • 沙箱是规则级原型，不是真实 TEE。
  • DP 同步是语义验证，不是完整预算系统。
  • intent gate 和 allowlist 仍是规则原型。

下一步

  • 扩大 gold set 和攻击查询集。
  • 接真实 OpenClaw memory search hook。
  • 补真实 sandbox runtime。
  • 做真实双设备或 Docker 双环境同步。
  • 增加人工效用评分和误杀/漏放分析。
稳妥表达：当前原型验证了治理机制方向有效，生产化还需要真实接线、真实沙箱、真实多设备和更大规模样本。
